# Design: Multiple / Tactical Callsigns

Generated by brainstorming on 2026-06-10

## Problem statement

Tuxlink models a single station identity: `IdentityConfig { callsign: Option<String>, identifier, grid }` in [config.rs](../../../src-tauri/src/config.rs), read at ~10 sites (CMS connect + B2F `mycall` in `winlink_backend.rs`, ARDOP/VARA `mycall` in `modem_commands.rs`, the ribbon DTO in `ui_commands.rs`, the consent gate). `Config::validate` enforces "callsign present iff `connect_to_cms`".

Operators need to operate under more than one identity:

- **Multiple licensed FCC callsigns** on one install (personal + club/trustee call; a shared station radio used by several licensed operators).
- **Tactical callsigns** — role/incident labels (EOC-3, AIDSTATION-1) operated alongside the licensed callsign.

This is a recognized EmComm need (WLE feature-inventory rows 7.5 "Multi-callsign install" and 7.6 "Add auxiliary callsign"); tuxlink is currently single-callsign.

## Requirements (operator-confirmed)

1. An identity **list**, **one active default at a time**. Two kinds: **FULL** (real FCC callsign) and **TACTICAL** (free-form label).
2. TACTICAL **rides under a parent FULL identity**: no own mailbox, no own credential — an "operate-as"/address-as label only.
3. Switching to a FULL identity **requires validating its credential against the OS keyring** — an anti-impersonation control. Possessing the machine must not let someone transmit under another operator's license (a Part 97 violation charged to the licensee). The FCC callsign is the **security principal**.
4. **Inbox is per-FULL-callsign** on disk. **Sent + Outbox are a shared master store, tagged by identity**; the Outbox is drained by the active session identity at send time (only that identity's queued messages go out).
5. A TACTICAL session is **blocked from CMS modes unless** tuxlink verifies the tactical address is registered with the CMS. P2P/RF modes are unrestricted.
6. The licensed FCC callsign always identifies the station on RF regardless of the active tactical label.
7. **Re-auth on launch** — the authenticated session is never persisted (see Security).
8. **Listeners are independent identity-bound sessions** (see Security).
9. Companion feature: **inbox routing rules** (tuxlink-73nl) sort delivered messages into folders; tactical mail routes into a per-tactical folder under its parent's inbox.

## How WLE handles this (verified from decompiled RMS Express)

Grounding from `dev/scratch/winlink-re/decompiled/RMS Express/` (per the project rule that prior-art implementations are ground truth for Winlink behavior, not the prose docs):

- **One active callsign at a time** — `Globals.strMyCallsign`, selected from a `cmbCallSign` dropdown.
- **A single "auxiliary" list** (`dicAuxCallsign`) holds *both* extra real callsigns *and* tactical addresses; same `DialogAddAuxCallsign`, and **each entry carries its own password**.
- The only difference between an aux callsign and a tactical address is a CMS validation branch: `IsTacticalAddress()` → `TacticalAddressExists()`; otherwise `AccountRegistered()`. **Both must be registered with Winlink** ("The tactical address … is not registered with the Winlink system.").
- Per-identity settings live in INI sections keyed by `CallsignAndQualifier()`.
- Inbound "is this for me?" uses one predicate, `IsOurCallsignOrTactical()`.
- **Mailbox is shared** (one message database; the active callsign is the send/receive-as lens + default sender).

**Where tuxlink deliberately diverges from WLE** (features yes, UX/assumptions no):

- **Tactical addresses are decoupled from CMS registration.** WLE requires every tactical address to be CMS-registered + globally unique because its tactical feature is CMS-routing-centric. Tuxlink supports P2P/RF, where a tactical label needs no registration and no global uniqueness — you simply *are* AIDSTATION-1 in that net. Registration is required *only* to receive CMS-routed mail to a tactical address (a CMS constraint, enforced at CMS-mode entry per requirement 5).
- **Per-callsign mailbox + authenticated switching instead of WLE's shared single-user mailbox.** WLE is a single-user Windows app; a shared mailbox + dropdown-only callsign switch lets anyone on a shared EOC/club machine read another operator's mail and transmit under their license. Tuxlink treats the FCC callsign as a security principal.

## Architecture: capability / handle model

Adopted after an independent Codex architecture evaluation (2026-06-10; raw transcript local-only at `dev/adversarial/2026-06-10-zjne-identity-arch-codex.md`, gitignored). Codex rejected a "thread an `active_identity()` config accessor through the codebase" approach because the anti-impersonation gate would be convention-only: `winlink_backend.rs` `set_config` can already swap live config, tests mutate `cfg.identity.callsign` directly, and a config pointer is forgeable by hand-edited JSON. The capability/handle model makes unauthorized transmit a **type error** instead.

### Components

- **`IdentityStore`** — persisted list of identities, **no secrets**.
  - `FullIdentity { callsign, cms_registered: bool, has_cms_account: bool, … }`
  - `TacticalIdentity { label, parent_full_id }`
- **`IdentityService::authenticate(full_id, credential) -> Result<IdentityHandle>`** — validates the credential against the OS keyring, then mints a handle.
- **`IdentityHandle`** — private fields, **in-memory only, non-`Serialize`**, constructible *only* inside `IdentityService` after keyring validation. Carries the authenticated `full_callsign`.
- **`SessionIdentity { handle: IdentityHandle, address_as: Address }`** — `address_as` is the FULL callsign or a tactical label whose `parent_full_id == handle.full_callsign`.
- **Active identity** — the default `SessionIdentity` for *new* compose/connect actions. Held in backend state, in-memory.

### Enforcement

- **Transmit + listen APIs take `&IdentityHandle` / `SessionIdentity`, never a raw `Config` callsign.** RF modem `MYCALL` always comes from `handle.full_callsign` (Part 97 station ID); the Winlink message `From` may use `address_as`. The ~10 current `cfg.identity.callsign` reads on transmit/connect paths are migrated to read through the session.
- A `TacticalIdentity` can only be bound into a `SessionIdentity` whose handle authenticates its `parent_full_id` — tactical cannot exist without an authenticated parent.

### The single biggest risk (Codex)

Conflating three distinct things — the **licensed station principal**, the **Winlink mail address**, and the **UI active selection** — into one mutable string. The handle model separates them: principal = `handle.full_callsign`, mail address = `address_as`, UI selection = a persisted "last selected" hint that is *not* authority.

## Security model

- **Authenticated switching.** Switching to a FULL identity calls `authenticate(full_id, credential)`; success mints the handle and sets it active. Possessing the machine ≠ possessing the identity.
- **No persisted authenticated session.** Persist only "last selected identity" as locked UI state. On launch, **re-auth is required** before any transmit, listen-arm, or Outbox drain under a FULL identity. The handle never touches disk.
- **Listeners are independent identity-bound sessions.** An armed VARA/ARDOP/packet listener captures its own `IdentityHandle` at arm time and answers under that identity until disarmed/expired. Switching the *active* identity (the default for new operations) does **not** mutate an armed listener. "Active identity" means "default for new operations," not "global principal for existing ones."

## Mailbox model

- **Inbox: per-FULL-callsign** — `mailbox/<callsign>/Inbox` (and per-callsign user folders). Mail addressed to a tactical label lands in its parent FULL's inbox, routed into a per-tactical folder by the routing-rules companion (tuxlink-73nl).
- **Sent + Outbox: a single shared store**, each message tagged with the identity it was sent/queued as. The **Outbox is drained by the active session identity** at send time — a session connected as W1ABC sends only W1ABC's queued messages. (Ties into the intent-filtered outbound drain in `winlink_backend.rs` touched by tuxlink-9efs.)
- The search index gains an identity tag; unread accounting stays consistent with `list()` (cf. tuxlink-mzm4) per destination folder.
- Mailbox UI gains an **identity filter**.

## CMS gating for tactical

A `SessionIdentity` whose `address_as` is a tactical label is **blocked from CMS modes** (Telnet-CMS, gateway-routed) unless tuxlink verifies that tactical address is CMS-registered. Verification is cached and re-checked online when possible; offline, an unverified tactical address may operate P2P/RF freely but not over CMS.

## Config / migration

- Replace the single-callsign `IdentityConfig` with an identity list (+ schema-version bump).
- **Migration:** the existing `identity.callsign` becomes the one FULL identity; the existing `native-mbox/inbox` becomes that FULL's inbox; existing Sent/Outbox messages get a default identity tag; the search index is rebuilt.
- The current `Config::validate` "CMS iff callsign" rule is replaced — it is false under tactical identities (a tactical operates without its own CMS account).

## UI (textual spec; mock during design-review)

**The switcher IS the existing callsign control in the top status bar** — not a new menu, not buried under clicks. Today that control is `.dash-callsign-row` (`data-testid="ribbon-callsign"`) in `DashboardRibbon.tsx` (the active callsign + its SSID select). It becomes the identity-switcher trigger, reusing pixels already on screen for space efficiency:

- **One click opens the list.** Clicking the active-callsign control in the top status bar opens a dropdown **anchored to it**, listing FULL identities with their TACTICAL labels nested underneath, a lock glyph on any identity needing authentication, and a per-tactical "CMS ✓ / blocked" badge. Picking an entry switches (or begins the unlock). No separate menu, no multi-step drill-down, no non-obvious path.
- **Authenticated switch, inline in the same dropdown.** Selecting a FULL identity that needs auth reveals an inline unlock field *within the open dropdown* (`Unlock W7XYZ  [password]  [Unlock]`) — no popup window. Access-control credential entry, distinct from the removed TX-consent modal (RADIO-1 governs transmission, not identity switching).
- When closed, the control keeps showing the active call (and its SSID) exactly as today, so the top bar's footprint is unchanged.
- **Mailbox identity filter** to view one identity's mail.

## Open questions (resolve in the implementation plan)

- **Activation secret for an offline / P2P-only FULL callsign** that has no CMS password. Likely a local passphrase set at add-time, bound in the keyring, so even non-CMS identities are gated. Confirm.
- **Keyring verification method** — compare entered password to the stored keyring value, or store a verifier (hash/derived key) so the raw secret need not be round-tripped for verification.
- **CMS-registration verification API + caching** for tactical addresses (which CMS query confirms a tactical address is registered; cache TTL; offline behavior).
- **Listener-session UI** — how armed, identity-bound listeners are surfaced per transport so the operator sees which call each is answering as.

## Out of scope / non-goals

- No added transmit safeguards beyond legacy WLE behavior (no airtime caps / TOT / per-transmission consent modal). The auth gate is **per-activation, not per-transmission**; RADIO-1 governs the transmission itself.
- No secrets on disk — keyring or in-memory only (`no-disk-creds-default`).
- Inbox routing rules are a **separate** feature (tuxlink-73nl), specced and built alongside but not in this doc's scope beyond the tactical-folder integration point.

## Testing strategy

- `IdentityStore` CRUD + tactical-parent invariants (a tactical must reference an existing FULL) — pure unit tests.
- `IdentityService::authenticate` — handle is minted only on keyring-validated credentials; rejects wrong/absent credentials; handle is non-`Serialize` (compile-time + a test asserting no Serialize impl path).
- Transmit/listen API signatures require a handle (type-level; a doc/test demonstrating a raw-callsign transmit does not compile).
- Outbox drain filters by active session identity (extends the tuxlink-9efs / tuxlink-mzm4 mailbox tests).
- Migration: a single-callsign config + existing mailbox migrates to one FULL identity with its inbox intact; search index rebuilds.
- CMS-mode gating: a tactical session is refused CMS mode unless registration is verified.
- Listener independence: arming a listener under identity A, then switching active to B, leaves the listener answering as A.

## Cross-model perspective (Codex, 2026-06-10)

Codex (read-only, high reasoning effort) verified the current single-callsign state against the repo and recommended **rejecting** the config-accessor approach in favor of the capability/handle model above, because only the handle model structurally enforces the Part 97 anti-impersonation requirement. It contributed the **re-auth-on-launch / never-persist-the-session** stance and the **independent identity-bound listeners** insight (switching the active identity must not mutate an armed listener). Its named biggest risk — conflating principal + mail address + UI selection into one mutable string — directly shaped the `handle` / `address_as` / "last selected hint" split.
