## Problem Statement

Tuxlink's first-run wizard (src/wizard/) configures the station — credentials,
location, VARA provisioning, test send — then drops the operator into the main
UI cold. Nothing teaches the surface map: ribbon Connect, mailbox, Contacts,
radio dock, Find-a-Station, Elmer. Flagship GUI apps ship an orientation layer;
Tuxlink ships only a configuration layer. New operators (Tech Prepper audience,
old-internet navigation habits) face a blank mailbox and an unexplained sidebar.

## What Makes This Cool

The tour registry doubles as a persistent spatial help layer. After first run,
the same spotlight machinery answers "where do I connect?" from Elmer chat —
the operator asks in words, the UI physically points. The 2026 "AI copilot
onboarding" pattern, using the copilot Tuxlink already ships. (Honest cost
note: the Elmer half is not "zero machinery" — it is one MCP tool PLUS a
backend→webview event bridge, an ack path, and a generated anchor catalog.
See Architecture. Plan phase must cost it as such.)

## Constraints

- WebKitGTK is the only render target that counts; overlay technique must be
  proven in the render harness (project has compositing/SVG-flex scars).
- ECT hard-support target + dependency posture: no heavy tour library.
- Emcomm reality: an operator opening the app mid-incident must reach work in
  one keystroke. The tour must never block, never nag, never re-fire.
- The tour points, it never fires actions — no stop triggers Connect/transmit,
  and the engine never navigates routes or opens panels on its own.
- AGPLv3 project; intro.js/shepherd AGPL terms are compatible but moot (not used).

## Premises (agreed; #3 revised per Codex challenge)

1. Wizard ≠ tour. Wizard completion only sets "eligible for first tour"; the
   app shell offers the tour when required anchors are mounted. Re-runnable
   from Help → "Replay tour". Re-running the WIZARD does NOT reset
   `onboarding.tour_completed`; only the operator's Replay does.
2. Never blocks: skippable at every stop; ESC dismisses the whole tour and
   COUNTS AS SKIP (`tour_completed` set — never offered again; only app-quit
   mid-tour without skip leaves the offer eligible once more). Completion
   persists to `onboarding.tour_completed`, tips to
   `onboarding.tips_seen: string[]` (see Config for the real setter shape).
   If the config write FAILS, an in-memory session flag still suppresses
   re-offer for the running session; the failure is logged (structured log,
   not console.debug) and the write retries on next state change.
3. REVISED — anchors are dedicated `data-tour-anchor` attributes (not
   data-testid, which is test plumbing); every step declares
   `requiredPanelState` and `fallback` so a collapsed/lazy-mounted/zero-rect
   target can never produce a spotlight on blank chrome. (Codex also proposed
   `requiredRoute`; dropped — the app has no router by design.)
4. Hand-rolled engine; no tour dependency. Honest size after review:
   ~800-1200 LOC including CSS, a11y, tracking, and the Tauri event listener.
5. No stop performs actions; highlighting Connect never clicks it. While a
   spotlight is active, the hole is covered by a transparent click-blocking
   fifth panel — the operator cannot click through mid-tour (stop 1 highlights
   ribbon Connect, whose click is the RF consent act; accidental click-through
   is unacceptable). Interaction resumes the instant the hint dismisses.
6. One hint engine serves the post-wizard tour, first-open tips, and Elmer
   point-at. Scheduling rules are separate from rendering. Elmer-fired hints
   NEVER mutate `tips_seen` (they are answers, not onboarding progress).

## Cross-Model Perspective (Codex, gpt-5.5 high, 2026-07-13 — verbatim)

1. "Coolest version: make the tour an 'Elmer can point at anything' system,
   not just first-run onboarding. After the first-run path, operators can ask
   'where do I connect?', 'where are forms?', 'how do I find RMS stations?'
   and Elmer triggers the same spotlight registry without taking actions.
   That turns onboarding into a persistent spatial help layer."
2. "Most likely to bite: 'anchor to existing stable data-testid attributes.'
   Test IDs are test plumbing, not UX contracts. Failure mode: the post-wizard
   UI has panels collapsed, lazy-mounted, hidden behind tabs, or resized by
   WebKitGTK differently; querySelector finds nothing or a zero-sized rect,
   and the spotlight points at blank chrome. Create explicit data-tour-anchor
   names and require each step to declare requiredRoute, requiredPanelState,
   and fallback." — ACCEPTED with one correction: `requiredRoute` is dropped
   as having no referent (this app deliberately has no router — src/routing.ts:
   two webviews, no react-router; see step schema).
3. driver.js gets closest (MIT, dependency-free) but "the missing 50% is
   Tuxlink-specific: persistence flags, React route/panel orchestration, safe
   non-action guarantees, WebKitGTK overlay tests, first-open scheduling, and
   an operator-grade content registry."
4. Build order: HintProvider React context (declarative, not imperative
   singleton) with startTour/showHint/dismissHint/resetHints; 4-panel dark
   overlay + positioned popover ("avoid clever SVG masks first; SVG masks and
   blend/compositing are exactly where WebKitGTK can surprise you") with
   measurement retry; registries at src/onboarding/tourRegistry.ts +
   hintRegistry.ts, outside src/wizard/.

Web landscape (2026): short tours (5-7 stops max), assistive/contextual
patterns preferred over big upfront walkthroughs, always skippable, AI-assist
emerging. All consistent with this design.

## Approaches Considered

### Approach A: Hint system only
HintProvider + overlay + tour/tip registries; 5-stop tour + first-open tips.
No Elmer integration (out of scope by operator decision, not deferred).
Effort S-M. Completeness 8/10.

### Approach B: Hint system + Elmer point-at (CHOSEN)
Everything in A, plus the `point_at` Elmer tool and its backend→webview
bridge, ack path, and generated anchor catalog. Effort M-L (honest costing
after review — the bridge is real plumbing). Completeness 9/10.

### Approach C: driver.js core + thin wrapper
Adopt driver.js for overlay mechanics; build registry/persistence/scheduling
around it. Fastest engine, adds a dependency and design-system styling
friction; Codex's missing-50% list is unchanged. Effort S. Completeness 7/10.

## Recommended Approach

**B.** The engine work is the same as A; the Elmer hook converts a one-shot
onboarding feature into a standing assistive layer — the same trajectory as
the voice-field-assistant thesis (the scarce operator asks, the UI points).

### Entry UX (exact)

On first arrival at the main UI with `wizard_completed && !tour_completed`,
a small NON-MODAL offer card appears bottom-right: "New here? Take a
90-second tour." [Start tour] [No thanks]. "No thanks" OR finishing OR
skipping mid-tour sets `onboarding.tour_completed = true`. Quitting the app
mid-tour without skipping leaves the flag unset → the OFFER (not the tour)
appears once more next launch, from stop 1. The offer card never blocks
input; the operator can ignore it and work.

### Components

- `src/onboarding/HintProvider.tsx` — context owning: active hint/tour state,
  persistence (config read/write + failure rule above), scheduling
  (post-wizard eligibility, first-open triggers, anchor-mounted gating), and
  the Tauri event listener for Elmer point-at.
- `src/onboarding/HintOverlay.tsx` — four fixed dark panels around
  `getBoundingClientRect()` + the transparent hole-blocking fifth panel + a
  popover card (title, body, step N/M, Back/Next, "Skip tour"). NO SVG masks.
  Position tracking: window `resize` listener + scroll listeners on ancestor
  scroll containers + ResizeObserver on the target + a short rAF-throttled
  post-render retry loop (ResizeObserver alone misses pure position changes).
  Popover collision handling: flip/shift to stay inside the viewport.
  Single-hint mode (first-open tips, Elmer point-at): no stepper/Back/Next,
  a "Got it" dismiss; ESC dismisses with no flag mutation beyond the tip's
  own `tips_seen` entry (point-at mutates nothing).
- Step schema (registry entries):
  `{ id, anchor, title, body, requiredPanelState, fallback }`
  where `fallback: 'skip' | 'center'` — `'skip'`: advance to the next stop
  and log; `'center'`: show the card centered with no spotlight. The engine
  applies the declared fallback when the anchor is missing or zero-rect after
  the retry window, and IMMEDIATELY (no retry window — probes are
  synchronous) when `requiredPanelState` evaluates false.
  `requiredPanelState` is a string key evaluated by a panel-state probe
  registry the AppShell registers with HintProvider (functions over shell
  state; HintProvider never reaches into other components' internals).
  Codex's generic `requiredRoute` field is DROPPED: this app has no router
  by design (src/routing.ts: two webviews, no react-router), all
  spotlightable surfaces live in the main webview, and surface visibility is
  entirely `requiredPanelState`'s job here.
- First-open tip trigger contract: the component hosting a tip's anchor
  calls `useFirstOpenTip(tipId)` when it becomes visible; HintProvider shows
  the tip iff it is not in `tips_seen` AND no tour/offer card is active
  (suppressed-not-consumed: an un-shown tip stays eligible). The Compose
  tip's host is the mailbox toolbar's Compose affordance (Compose itself is
  a separate window) — it fires on the first mailbox visit with no tour
  active.
- `src/onboarding/tourRegistry.ts` — the 5 first-run stops (draft):
  1. Ribbon Connect — "One click runs your last-configured session."
  2. Mailbox — folders, unread, where mail lands after a connect.
  3. Contacts — the one address surface (All / ★ Favorites / Heard).
  4. Radio dock — modes, arming a listener, session status.
  5. Elmer — "Ask anything. Try: where do I connect?" (bridges to point-at).
- `src/onboarding/hintRegistry.ts` — first-open tips: Find-a-Station, APRS,
  Settings, and Compose. The Compose tip anchors on the MAIN-WINDOW
  affordance that opens Compose — Compose itself is a separate Tauri window
  (`compose_window.rs`) outside the main window's React tree; no provider is
  mounted there and nothing in that window is spotlightable in v1.

### Accessibility & keyboard

- Popover is `role="dialog"` with `aria-labelledby`/`aria-describedby`;
  stop advancement announced via an `aria-live="polite"` region naming the
  spotlighted control.
- Focus is trapped in the popover while a hint is active; Tab cycles its
  controls; focus returns to the previously-focused element on dismiss.
- ESC dismisses the tour; ←/→ or Enter advance/go back.
- App-global keyboard shortcuts are suppressed while the overlay is active.
  Mechanism (the app has 15+ scattered document/window keydown listeners and
  no central shortcut manager): a single CAPTURE-PHASE `window` keydown
  handler registered at app boot — before all other listeners — that swallows
  events (except the tour's own keys) while `overlayActive` is true and is a
  no-op otherwise. The plan phase enumerates existing WINDOW-level
  capture-phase keydown registrations to verify none precedes the boot
  handler (document-level sites are structurally safe — capture runs
  window → document → target). The overlay unmounts fully when idle, so a
  working operator's shortcuts are untouched.
- Theme: overlay + popover use design-system tokens only, re-render on theme
  change; render-harness fixtures in BOTH default dark and light.

### Elmer point-at (the bridge, named)

- MCP tool `point_at { anchor_id }` on the in-process transport
  (src-tauri: elmer executor → rmcp router, same registration path as
  existing tools).
- The tool handler validates `anchor_id` against the anchor catalog; on
  unknown ID it returns the list of valid IDs (Elmer self-corrects).
- Delivery: `app_handle.emit_to("main", "onboarding://point-at", payload)`.
  HintProvider (main window) listens and calls `showHint(id)`.
- Ack: the frontend invokes `onboarding_point_at_ack { request_id, outcome }`
  where outcome ∈ {shown, anchor-unmounted, overlay-busy}; the tool awaits
  the ack with a 2 s timeout. Timeout/negative outcome → the tool returns a
  structured failure so Elmer never claims "I've highlighted it" while the
  operator sees nothing. On `anchor-unmounted`, the tool's error includes the
  catalog's "how to open this surface" line so Elmer can instruct the
  operator (the engine NEVER navigates on its own).
- Anchor catalog: GENERATED at build time from tourRegistry + hintRegistry
  (script emits Elmer's knowledge-tier doc) — one source of truth, no drift.
  DEPENDENCY: the Elmer knowledge tier is itself spec'd-but-unbuilt
  (docs/superpowers/plans/2026-07-13-elmer-knowledge-tier.md); the plan phase
  records a bd dep edge. Degraded mode if point_at ships first: the tool's
  description plus its unknown-ID error (which returns the valid-ID list)
  lets Elmer self-serve anchor discovery without the catalog doc.
- Crate boundary note for planning: `point_at` crosses the mcp-core boundary
  like every tool — new port trait in tuxlink-mcp-core/src/ports.rs, adapter
  in src-tauri/src/mcp_ports.rs holding the AppHandle plus a pending-ack map
  (request_id → oneshot) in managed state. Cost it explicitly.

### Config (the real persistence shape — there is no generic config_write)

Tuxlink persists config through SECTION-SCOPED setters (`config_set_privacy`,
`config_set_grid`, … in src-tauri/src/ui_commands.rs), and `config_read`
returns a `ConfigViewDto` projection. This feature therefore adds:

- `OnboardingConfig { tour_completed: bool, tips_seen: Vec<String> }` in
  src-tauri/src/config.rs with `#[serde(default)]` (config uses
  deny_unknown_fields).
- **CONFIG_SCHEMA_VERSION bump 6→7** — additive fields without a bump made
  older binaries reject the whole config once before (the trash_auto_purge
  incident); the `config_schema_version_tracks_field_set` test enforces this.
- New fields on `ConfigViewDto` (the projection-omission class has also
  bitten before: map_tile_source).
- One setter command `config_set_onboarding` (read-modify-write under the
  config mutex); the premise-2 retry-on-failure rule rides this setter.

**Upgrade cohort rule:** a pre-v7 config with `wizard_completed: true`
migrates with `tour_completed` left false — existing operators see the
one-click offer card exactly once (they may take the tour if curious) but
never get first-open tips on surfaces they have used for months. Fresh
profiles get both. Mechanism (tip IDs live in the frontend; no Rust-side ID
duplication): the MigrateAdditive path, when loading `schema_version < 7`
with `wizard_completed: true` and no `onboarding` section, seeds the
sentinel `tips_seen: ["*"]`, which `useFirstOpenTip` treats as all-seen.
Fresh-profile restarts post-wizard are unaffected (their config is already
v7 with a real `onboarding` section).

Help menu gains "Replay tour".

## Success Criteria

- Fresh profile: wizard completion → offer card exactly once; "No thanks" →
  never again; accept → 5 stops; quit mid-tour → offer once more, from stop 1.
- Each deeper surface shows its tip exactly once, first open only.
- Asking Elmer "where do I connect?" spotlights ribbon Connect within 2 s or
  Elmer receives a structured failure (never a silent no-op).
- Keyboard-only traversal of the whole tour works; screen reader announces
  each stop; focus restores on dismiss.
- All overlay states rendered + approved via the WebKitGTK render harness
  before merge (fixtures: dark + light, normal + compact viewport, fallback
  'center' state, offer card).
- Zero regressions while overlay inactive (engine unmounts fully; no
  permanent portal layer; global shortcuts unaffected).
