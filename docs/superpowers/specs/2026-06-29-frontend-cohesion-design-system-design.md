## Problem Statement

Tuxlink's frontend (Tauri 2 + React 18 + TS, WebKitGTK-rendered) is broadly modern and
professional, but specific surfaces read as "AI-generated" in a negative way. The owner
can feel it but struggled to translate the feeling into actionable direction. Two named
surfaces:
1. **Radio panes** (Packet AX.25 / VARA / ARDOP), full manual mode — buttons and fields
   too wide; groupings/sizing a human wouldn't choose.
2. **Main status bar** — every control styled differently (built at different times),
   reads as disjointed.

Explicit constraint: **NOT a redesign.** The look is mostly liked. Fix the weak areas.

## Diagnosis — "AI-generated" translated to named, located defects

The feeling is real signal; these are the mechanical causes (evidence = file:line):

1. **Over-wide buttons = `.radio-panel-btn { flex: 1 }`** (`src/radio/RadioPanel.css`).
   `flex:1` forces every button in a row to stretch and split width equally regardless of
   label. That is the single biggest "AI did this" tell in the radio panes. A human sizes a
   button to its content and gives the primary action visual weight.

2. **Disjointed status bar = no shared control scale.** Adjacent `dash-*` controls
   (`src/shell/AppShell.css`, `src/shell/StatusBar.css`) use font-sizes of **9px / 13px /
   14px**, paddings of `0`, `0 4px`, `1px 6px`, `0 16px 0 4px`, and borders of `none` /
   `1px transparent` / `1px solid`. Each control picked its own values. That is the absence
   of a system, not a system.

3. **Root cause = no shared control primitives + scattered tokens.** No `Button`/`Input`
   React component, no `src/components/`. Design tokens (CSS custom properties) are defined
   in ~5 separate files (`src/App.css`, `src/shell/StatusBar.css`, `src/wizard/wizard.css`,
   `src/mailbox/MessageView.css`, `src/help/HelpView.css`). Every control re-decides its own
   padding/height/radius/font-size. The radio panes DO share a `radio-panel-*` vocabulary
   (good) — their problem is the sizing defaults inside it, not the absence of it.

## The "how do I give direction" reframe

The owner does not need to translate the vibe into a spec. The conversion is a repeatable
loop, not a talent:
1. Render the surface — `dev/render-harness/` produces WebKitGTK-accurate PNGs, no Rust build.
2. A vision model names the tells against the CSS, with file:line ("buttons stretch via
   `flex:1`", "three font sizes in the bar").
3. Each named tell becomes a directive.
4. Fix, re-render, diff the before/after PNG as proof.
The owner's "this looks off" is the *input* the loop needs; the loop produces the precision.

## Premises

1. The look is mostly good; this is targeted consolidation, not a redesign. — agreed
2. The two visible symptoms share one root cause (no shared control layer + scattered
   tokens); fixing the root fixes both and prevents recurrence. — agreed
3. A big-bang restyle is the real risk (hours of remediation / full revert); phasing is the
   risk control. — agreed (owner: "C, but ideally phased")
4. WebKitGTK is the only render target; correctness is judged on render-harness PNGs, not a
   Chromium proxy. — agreed (per project memory)

## Approaches Considered (control-layer composition)

### Approach A: CSS tokens + shared utility classes only
Smallest per-phase diff (CSS-only, simplest reverts). Does not hard-prevent a future bespoke
control; a stylelint rule on raw values is the only guard.

### Approach B: React control primitives everywhere
Strongest enforcement (the component is the only way to make a control), but every surface
migration edits JSX — bigger diffs, more revert surface per phase. Heaviest churn.

### Approach C (CHOSEN): Hybrid — CSS tokens/classes + thin React wrappers for the big-3
Tokens + shared CSS classes are the additive, zero-risk foundation (CSS-only diffs per
surface = easy to verify + revert). Promote the 3 highest-reuse controls (button, select,
field) to thin React wrappers so new code physically can't reintroduce `flex:1` or a one-off
font-size. A stylelint guard bans raw size/spacing literals outside tokens. Comprehensive
consistency + drift-prevention, still phased and per-surface revertible.

## Recommended Approach — phased execution

**Scope:** full design-system consolidation (every surface), executed phased so any single
phase is independently shippable AND revertible. The foundation is additive (changes nothing
on its own); each surface migrates in its own PR with a before/after render-harness PNG.

**Phase 0 — MINIMAL foundation, scoped to the first pilot (Codex de-risk):** do NOT build the
whole abstract design system up front. Build only what the status-bar pilot needs:
- The token file with the scale above (spacing/control/type/radius) — but only the tokens the
  pilot consumes need to be exercised; reconcile with existing color tokens.
- The shared CSS control classes the status bar uses (`.tux-btn`, `.tux-field`, `.tux-select`).
- stylelint in **warn/baseline** mode (not error) so it doesn't block while the scale settles.
- Capture render-harness baselines (at least the pilot surfaces).
- Do **NOT** freeze the React-wrapper (`Button`/`Select`/`Field`) API yet.

**Phase 1 — Status-bar pilot (first visible win + WebKitGTK proof).** Migrate `dash-*` onto the
scale; re-render; diff PNG. This is where the foundation meets real WebKitGTK rendering.

**Phase 2 — Radio panes.** Drop `flex:1`, size-to-content, primary emphasis; re-render; diff.

**→ Freeze the big-3 wrapper API + flip stylelint to error ONLY after Phases 1+2 both survive
screenshot review.** Two real surfaces validate the primitives before they're locked.

**Phases 3..N — remaining surfaces, worst-first** (now including Codex's finds): Wizard
(gradient + animation + radius soup — the loudest generated-UI signature) → the duplicated
dialog chrome (Catalog/GRIB/FormPicker → one `.tux-dialog`) → Compose → Settings → Forms →
Sparkline (candy gradients) → SessionLog glyphs. Each: migrate, re-render, diff PNG, ship.
Per-phase cross-provider: Claude visual pass on the PNG + Codex CSS read, reconciled. A botched
phase reverts one surface, never the app.

## Token Scale (starter, from Codex; reconcile with existing `--accent`/color tokens in Phase 0)

```css
:root {
  --space-1: 2px; --space-2: 4px; --space-3: 6px; --space-4: 8px;
  --space-5: 12px; --space-6: 16px; --space-7: 20px; --space-8: 24px; --space-9: 32px;
  --ctl-h-xs: 22px;  /* badges, statusbar controls */
  --ctl-h-sm: 26px;  /* compact buttons/selects */
  --ctl-h-md: 30px;  /* default inputs/buttons */
  --ctl-pad-x-xs: 6px; --ctl-pad-x-sm: 8px; --ctl-pad-x-md: 10px;
  --type-micro: 10px; --type-meta: 11px; --type-control: 12px;
  --type-body: 13px; --type-heading: 16px;
  --radius-xs: 2px; --radius-control: 3px; --radius-panel: 6px; --radius-pill: 999px;
}
```
The control heights (22/26/30) and the 10/11/12/13/16 type scale match what the codebase
*already* mostly uses (status bar 9-14px → snaps to micro/meta/control/body; radio panes
already 11-14px), so adoption is mostly snapping outliers to the nearest token, not a
restyle.

## Success Criteria

- Status bar: one font-size scale, one padding rhythm, one border treatment across all
  `dash-*` controls; before/after PNG shows visible cohesion.
- Radio panes: buttons sized to content with clear primary emphasis; no `flex:1` stretch.
- New controls added after Phase 0 use the wrappers/tokens by default; stylelint blocks raw
  values.
- No phase changes more than one surface; each phase reverts cleanly in isolation.
