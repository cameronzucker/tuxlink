Tuxlink — GPS setup UX brainstorm

2026-06-04 • magpie-isthmus-gorge • backed by dev/research/2026-06-04-winlink-group-pain-points.md

Reading guide

Four wizard-step states (success, native-NMEA, triage, manual fallback) + one Settings GPS panel for the post-setup surface. All callsigns and grids are realistic placeholders. Sudo commands are copy-paste-ready.

Design principle: "failure mode is the product". The interesting UX work is what we do when GPS doesn't work, not when it does.

Anchors: v0.0.1-ux-principles.md §1 ("intuitive like Thunderbird") + P3 ("promote dashboard info legacy Winlink hides"). The four personas below ground the design — each gets a coherent path.

The four personas hitting GPS setup

Frequency in the wild is rough estimation. The EmComm field op is the highest-stakes — the others have time to recover from a stumble; she may not.

Bob the seasoned Debian ham K6ABC
Fixed home station. Linux for 15 years. Already runs gpsd because xastir wants it.
  • Outcome on first run: tuxlink finds gpsd silently, fix is live, "use this" pre-selected. One Enter and he's done.
  • He's the easiest. He's also the loudest if we make him click through 5 screens to confirm what he already configured.
  • What we owe him: a wizard step he can blow past in under 2 seconds.
Sue the EmComm field operator W7SUE
ARES deployment, parking-lot QTH. Field laptop, no gpsd configured, USB GPS puck on the table.
  • Outcome: tuxlink notices the USB GPS, reads NMEA, offers the device by name — without her installing or configuring gpsd.
  • If we make her edit /etc/default/gpsd with the wind whipping the EOC tent flap, we have failed her.
  • What we owe her: zero-config recognition AND a fast manual-grid fallback if the GPS won't cooperate.
Dave the Windows-Winlink convert N3DAV
25 years of Winlink Express. Just installed Mint on a refurb ThinkPad. First Linux ham app he's ever tried.
  • Outcome: he plugs in his GPS, nothing happens, the wizard shows him why and what to type. The fix is 60 seconds and a re-login.
  • If the failure surface says "GPS not detected. Check your configuration." — he uninstalls tuxlink and posts in the Winlink group about Linux being broken.
  • What we owe him: every error includes the exact command. dialout, ModemManager, none of it should require Google.
Mike the fixed-station retiree W4MIKE
FT-991 + dipole + log table. Knows his grid. No GPS, doesn't want one, runs Pop!_OS on his shack PC.
  • Outcome: one click to skip GPS entirely, manual Maidenhead entry with sane defaults.
  • If we force him to wait through a 30-second gpsd timeout before he can type "EM35vx", we've already lost his trust.
  • What we owe him: the "I don't have GPS" path is a first-class option in the wizard, not buried under "Advanced."
Design implication: the wizard step has FOUR states the user can land in, and the LAYOUT of each state should make the user's path one click. The detection probe runs in parallel with the manual-entry option being visible — no one waits for a timeout to skip.

Step 4 of 6 — Location

No GPS detected. Here's what's getting in the way.

Tuxlink looked in three places. Each one had a specific problem — most of these take one command to fix.

✗

You're not in the dialout group.

Linux requires this group to read GPS / serial devices. We can see /dev/ttyACM0 on your system but can't open it.

"Fix it for me" pops a system password prompt (PolicyKit), runs usermod -aG dialout $USER, then asks you to log out and back in. Linux loads group membership at login — that step's mandatory, not ours.

⚠

ModemManager may be hijacking your GPS.

ModemManager runs AT commands at every USB serial device, including GPS pucks. It can wedge them. If you don't have a USB cellular modem, you don't need ModemManager.

PolicyKit prompt → systemctl mask ModemManager. Reversible from the Settings → Location → Troubleshoot panel if you ever need cellular USB modem support.

○

No gpsd on this system.

Not required — once dialout is fixed, tuxlink reads the GPS directly. If you'd rather share it with other ham apps, install gpsd.

Optional. Opens a system terminal with apt install pre-typed — apt is interactive so we don't run it silently.

Post-setup — when something changes (field deployment, distro upgrade, device swap)

Cameron's pushback (2026-06-04): the wizard runs once, but stations get stood up in one place and deployed somewhere else. Sue packs her laptop at home (gpsd happy) and arrives at an EOC parking lot (no gpsd, USB puck plugged in). Dave's distro upgrades and ModemManager comes back from the dead. Bob upgrades from a u-blox 7 to a u-blox 9. The Settings panel needs the same triage UX the wizard had, plus a clean way to switch sources without restarting.

Architectural answer: the wizard's GPS step and the Settings → Location panel render the same component. Different chrome (wizard has Back/Continue; Settings has Save/Cancel), same detection + triage + "Fix it for me" surface. The user who learned the wizard knows Settings by inference.

The "Troubleshoot" tab/section is also reachable manually from Settings → Location even when things look fine, in case the user wants to inspect.

Locked decisions (Cameron, 2026-06-04)

✓ First slice = full UX, not phase-gated

"Fix it for me" (pkexec) ships alongside "Show command" in the same release. Rationale: shell friction is enormous even for technical users — see Cameron's Laserfiche Support Tools, a product that exists because his engineers wouldn't paste commands. "Give a shit" means respecting shell friction. Power users keep the copy-paste escape hatch; the rest get the magic button.

✓ Tuxlink never holds sudo; PolicyKit owns the trust surface

"Fix it for me" invokes pkexec /usr/libexec/tuxlink-gps-fix <action>. The helper is a tiny binary shipping a fixed set of actions (add-dialout, mask-modemmanager, install-gpsd). A PolicyKit policy in /usr/share/polkit-1/actions/com.tuxlink.app.policy authorizes only the helper, not arbitrary commands. The user types their password in a system-rendered dialog and consents to one specific action.

✓ The "whoa" moment we polish hardest

Dave's path. The triage panel that detects ModemManager + dialout + missing-device with the exact diagnosis, and the "Fix it for me" button that just works. That's the screenshot a tuxlink user posts to their club Slack. Bob and Sue get their happy paths along the way, but Dave's path is what differentiates this from every other Linux ham app.

✓ Wizard step and Settings panel share one component

The GPS source picker + triage UX is a single React component mounted in two contexts. Wizard wraps it in Back/Continue chrome; Settings wraps it in tabs (Troubleshoot / Source picker / Manual grid / Privacy) plus a Save/Cancel footer. Detection probes, source cards, triage cards, "Fix it for me" buttons — all shared. Field deployments and distro upgrades are first-class flows, not afterthoughts. The dashboard pill is the always-visible "tuxlink noticed something" surface that bounces the user into the relevant Settings tab.
