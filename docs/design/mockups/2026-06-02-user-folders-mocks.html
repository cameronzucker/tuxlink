User folders — unified mechanism brainstorm

Combines tuxlink-ca5x (Archive wiring) + tuxlink-f62f (custom folders) into one design. 2026-06-02 · pika-cedar-tanager.

Premise

Archive and custom folders are the same shape: a movable mailbox destination. The backend already supports an archive folder end-to-end (see native_mailbox.rs::move_to); only the frontend renders it disabled. Custom folders extend the same mechanism: open-set folder names with create/rename/delete UI.

What this mock decides: sidebar placement (3 variants), folder ops surface area (context menu + dialogs), and message-move affordances. Backend storage model is sketched in the spec but is not the visual question here.

Sample mailbox: realistic NW Oregon ARES traffic — ICS-213, weekly net check-ins, NTS, position reports. Callsigns are the operator's actual rotation (KE7VAR, W7AAR, KD7HBA, N7ALT) lightly fictionalized.

Decisions the operator needs to settle

Each row has a recommendation; brainstorming converges by either accepting or redirecting each one.

IDQuestionOptions
D1 Sidebar placement A) Flat list under Mailbox · B) Separate "Folders" section · C) Archive system, custom in folders
Recommend B — Connections section already uses the pattern; + button stays visible; unifies the user-folder mental model.
D2 Archive: system or user folder? Pre-seeded user folder (renamable, deletable) · System folder (always present, undeletable)
Recommend user folder — the backend already has it as a string-named directory, not a special table. Reserve the name "Archive" so a new install always has one, but treat it like any other user folder for ops. A user who deletes Archive can recreate it.
D3 Folder structure: flat or nested? Flat (Winlink Express convention) · Nested (Outlook style)
Recommend flat for v1 — Winlink Express convention; matches Cameron's preference; nesting can be added later if asked.
D4 Backend storage <root>/<folder-slug>/<mid>.b2f — directories as folders (matches today) · Separate user_folders.json metadata index · SQLite table
Recommend directories + metadata sidecar. Folder = directory; <root>/.folders.json maps slug → display-name + creation-time. Rename = update JSON (slug stays); delete = empty dir + remove JSON entry. archive/ is a default entry in .folders.json at first launch.
D5 Move semantics Single-message context menu · Multi-select · Drag-drop · Keyboard shortcut (A for Archive)
Recommend context menu + A-shortcut for v1. Multi-select + drag-drop deferred. The Archive button in the reading-pane toolbar is the one-tap path; Move to submenu is the deliberate path.
D6 Delete non-empty folder Refuse with error · Prompt: move to Inbox / Archive / delete · Cascade silently
Recommend prompt (mock above shows the dialog). Inbox is the safe default; user explicitly opts into "delete the messages too."
D7 Folder name constraints Reserved: Inbox, Sent, Outbox, Drafts, Archive (Archive only if it doesn't already exist as a user folder) · Case-insensitive uniqueness · 3-40 chars · Allowed: letters, numbers, spaces, dashes (no slashes, no leading dots)
Recommend as listed. Slug = lowercase + hyphenate spaces; display name preserves case + spaces.
D8 Search interaction folder:archive token works · folder:ares-drills resolves on slug · Quoted multi-word: folder:"ARES Drills"
Recommend slug lookup + quoted display-name lookup. Already supported by ChipStrip's FOLDER: token; index needs to learn new folder names dynamically.
D9 Unread badge on user folders? Only Inbox shows unread (today) · All folders show unread · All folders show total instead
Recommend total count for user folders — matches Sent today; user-archived messages are almost always already read, and showing "0 unread / 180 total" on Archive would be misleading. Match Sent's behavior.
D10 Compose: target folder? Drafts → user-folder on send · Drafts only (current behavior, defer "save to user folder") · Allow user to choose target folder before send
Recommend defer. Sent goes to Sent; saves go to Drafts. Moving drafts to user folders is a v1.5 concern. Don't over-design.

Implementation sequence (if Concept B + the rec column are accepted)

Phase 0 (this brainstorm): ship this mock + spec, settle decisions D1–D10, write the spec doc under docs/superpowers/specs/2026-06-02-user-folders-design.md.

Phase 1 — tuxlink-ca5x (Archive only): small. Frontend: add 'archive' to MailboxFolder + BACKEND_FOLDERS; flip the disabled flag; wire a Reading-pane Archive button + A shortcut that calls mailbox_move(from, 'archive', mid). Backend: thin move_message Tauri command wrapping the existing Mailbox::move_to. No new storage. ~1 day, ~10 files.

Phase 2 — tuxlink-f62f (open-set folders): the architectural work. UserFolder registry (.folders.json), folder_create/rename/delete commands, dynamic MAILBOX_ITEMS, Move-to submenu, new-folder dialog, folder context menu, delete-confirmation dialog. Slug validation, reserved-name list, search-index dynamic folder names. Tests for the full lifecycle. ~3-5 days.

Phase 3 — deferred (file separately if wanted): multi-select move, drag-drop move, nested folders, draft-to-folder, folder reorder.
