Post Office modes — final UX

tuxlink-6c9y. Routing follows the connection, and you pick what to send at send time. No compose-time choice, no footgun.

Three distinct session types

These two modes join the existing Winlink (CMS) mode. They differ on two axes — how the client connects, and what the relay does with the mail. Network Post Office sits in the middle.

Session typeHow it connectsWhat happens to the mailWhen you'd use it
Winlink (CMS)direct to the CMS (internet / RF gateway)enters the global Winlink systemthe CMS is directly reachable
Network Post Officeto an RMS Relay on a LAN / AREDN mesh (full callsign, C)normal routing — the relay forwards onward, and can deliver to local mesh recipientsa relay is reachable but the CMS is not
Telnet RMS Post Officeto an RMS Relay, typically local (CALL-L, L)held in the relay's local pool for local pickup — never forwarded globallylocal delivery that stays off the global system
The divergence (front-and-center in docs): tuxlink drops WLE's compose-time "message type" / routing pools. The relay decides routing from the login (a Telnet RMS Post Office logs in as CALL-L); the message bytes are identical. You open the connection that means what you want, and select which Outbox messages to send. tuxlink still emits the same X-RMS-Routing: L header WLE does, derived from the connection at send time — so it stays byte-identical and fully interoperable on a shared relay.

Compose — unchanged

No new control. The deferred "Send as" field stays disabled at "Winlink Message". Composing says nothing about routing — that's decided when you connect.
