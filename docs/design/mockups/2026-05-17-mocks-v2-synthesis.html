Synthesis — B as default + togglable C-style radio dock

Showing the agreed direction in two states: default (dock off, like B) and with View → Show Radio Dock toggled on. Same window, same content, one toggle changes the layout.

Synthesis decisions captured from your "between B and C" call

  • B's structure wins as default: dashboard ribbon up top (callsign · grid · GPS · UTC+local · connection); 3-pane below (sidebar · rows · reading); human-shaped session log default-visible with raw toggle; minimal status bar; separate-window compose.
  • C's radio dock becomes a togglable 4th pane via View → Show Radio Dock (Ctrl+Shift+R-conflict; pick a different shortcut). Default: hidden. For v0.0.1, the dock shows what telnet-only backends actually surface (session timer, sent/received counters, outbox queue, last-session log). v0.1+ unlocks: TX/RX state, audio levels, GPS sats, battery.
  • Dashboard ribbon stays compact in v0.0.1 (5 items). Heavier emcomm signals from C live in the radio dock, not in the always-on ribbon. Keeps the default surface clean for the Mail.app-style user; emcomm power-users opt in.
  • Compose is a separate Tauri window (Mock B's choice). This requires expanding plan Task 14 to multi-window from its current Dialog-only spec — surface this as a plan amendment in the design doc.

Dock-on notes: radio dock claims ~220 px on the right; rows + reading pane shrink slightly but stay usable on 1366×768 (sidebar 180 + rows 340 + reading ~600 + dock 220 = ~1340 + window chrome). For v0.0.1, dock contents are scoped to what telnet-only backends provide (current session timer, outbox queue, last-session log). v0.1+ unlocks the radio/audio/GPS/battery section currently shown as v0.1 stubs. Showing those stubs IS deliberate — gives the dock its identity and signals to the operator "more is coming."

If this matches your "between B and C" call, next steps for the brainstorm:

  • Confirm scope of v0.0.1 dock contents — exactly which sections in v0.0.1 (current session / outbox / last sessions / radio-audio-v0.1-stubs), or scope it tighter? The Last Sessions list, for instance, needs backend work the plan doesn't currently include.
  • Confirm compose-as-separate-window as an amendment to plan Task 14 — moving from Radix Dialog to a separate Tauri window is a real scope addition; surface it as a plan-amendment line item in the design doc.
  • Wireframe the wizard screens (Tasks 9-11) in this same dark-mode polish — first-run is the user's first impression and the design doc explicitly weighs it heavily (P5: "zero genuine user barriers on first-run").
  • Compose-window wireframe — a separate small dark mock showing the floating compose window in detail.
  • Error / empty / connecting states — the existing mocks all show "happy path"; emcomm needs the failure-mode UX too (CMS unreachable, credentials rejected, mid-session disconnect).

Tell me in chat which of these you want to drill into next, or flag what else still needs reaction at this layout-level before moving on.
