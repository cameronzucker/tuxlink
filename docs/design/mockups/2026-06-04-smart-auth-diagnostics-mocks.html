Mock conventions

Each section shows the Smart Auth-Failure Diagnostic Banner pinned to the Telnet modem dock, above the session-log section. The dock width is the production-CSS 400px (matches .radio-panel in src/radio/RadioPanel.css). The session-log section is collapsed to a one-line stub in these mocks to focus attention on the banner. All copy is tuxlink-original (Q3) — the WLE wording cross-walk appears in section H.

Uncategorized fallback

Any `*** ...` payload not matching Modes 2–4. Banner uses warning yellow (not red) — we don't know what's wrong, but the user shouldn't be alarmed about a credential or callsign problem when we can't actually confirm one.

Open questions for adrev to weigh

  • Banner pin point: the banner sits between Transport and Session Log sections. An alternative is to pin it at the very top of the panel body (above Server). Pro: more discoverable. Con: pushes Transport/Server controls down on every successful connect, which is most connects.
  • Raw-response default state: collapsed by default with click-to-expand. Alternative: expanded by default for Modes 2 + uncategorized (where the wire string IS the diagnostic information). The mock shows the expanded variant for those modes.
  • External-link confirmation: "Reset on winlink.org" opens the browser immediately. Should there be a confirmation step ("This will open your default browser to winlink.org. Continue?") to avoid surprising users? Default proposal: no — modern apps don't gate this.
  • Test-credentials latency: the "Testing…" state can take 5-30 seconds depending on network. Should there be a progress bar or a more elaborate spinner? Default: subtle `⟳` glyph; the session-log timestamps show progress.
