Reset on previous mocks

v1 (mocks A/B/C) had an in-panel "Dial as: CMS / P2P" toggle. That's a category error: the tuxlink sidebar already picks (sessionType, protocol), which is the same role WLE's cmbSessionType dropdown plays. PR #348's ARDOP intent toggle was wrong for the same reason. Both panels (ARDOP + VARA) should read intent from RadioPanelMode props and render accordingly. v2 rebuilds against the WLE-grounded model.

WLE reference fidelity (what we adopt; what we don't)

Adopt: single Start action (Connect + B2F + Disconnect as one bounded op, one RADIO-1 modal); target + bandwidth picked BEFORE Start; auto-managed transport (no separate "Open VARA TCP" step); per-intent routing-flag enforcement on outbound drain.

Don't adopt for alpha: frequency displays (no CAT integration in tuxlink today); favorite channels combo (cmbFavoriteChannels — useful but post-alpha); WLE's bearing/distance/quality toolbar readouts (defer to ARDOP signal panel pattern).

Tuxlink divergence: radio-only gets a listener (WLE doesn't). Operator opts in by clicking the "Radio-only" sidebar entry under VARA HF / VARA FM / ARDOP HF.

Notes:
· Panel chrome is shared across all three intents: same component, same header, same session log, same auto-managed transport at the bottom. The intent pill in the header makes the chosen mode self-evident. The middle column shows an armed P2P listener with the Start button disabled (modem busy — one operation at a time).
· The Start button drives the whole outbound flow: one click → RADIO-1 modal → backend mints a token → CONNECT → B2F exchange → DISCONNECT, all under one bounded-airtime cap. Matches WLE's mnuStart behavior in the session form. No separate "Connect" + "Send/Receive" affordance.
· Listener and outbound are mutually exclusive in time: modem can only run one ARQ session at once. If listener is armed and operator clicks Start, the button is disabled with a "disarm to dial out" hint. If outbound is in flight and operator tries to Arm, same idea.
· ARDOP gets the same rewire: PR #348's "Dial as" toggle goes away. ArdopRadioPanel reads intent from props; sidebar drives. parse_b2f_intent widens from cms|p2p to cms|p2p|radio-only.
· Modem expander at panel bottom: host/port (VARA) or audio devices + binary path (ARDOP). Collapsed by default. Auto-opens transport on first Start/Arm click. No Start/Stop buttons for the transport itself.

Notes:
· Same intent throughout. The panel header doesn't switch tabs or modes; only the status pill + action surface changes.
· Abort semantics matter: during dial, "Abort" sends DISARM to the modem (kills CONNECTING) — cheap. During exchange, "Abort + Disconnect" sends ABORT (force-end ARQ session) + DISCONNECT — what the tuxlink-12sc follow-up addresses for VARA's listener path. Same wire-form for outbound + inbound.
· Auto-return to idle: after exchange completes (success or fail), session returns to idle with a one-line outcome banner ("Last exchange OK: 2 sent, 1 received"). Operator can re-Start with same or different target.
· Bandwidth selector reappears in idle but is disabled mid-exchange (modem won't honor a change anyway).

Still-open shape decisions (we'll work through these one at a time)

  • RADIO-1 consent modal copy: the modal text should reflect the chosen intent (cms vs p2p vs radio-only) so the operator's confirmation is informed. Same modal component, intent-aware messaging.
  • Frequency / dial display: WLE has it, tuxlink has no CAT. Add operator-typed informational field for alpha or defer entirely?
  • Favorite Channels: useful for fast re-dialing common targets, but feels post-alpha. Skip for v1?
  • Per-intent outbox filter visibility: should the panel surface "0 C-tagged messages in outbox" to warn the operator before they dial a CMS session that won't drain anything?
  • ARDOP-specific transport expander: audio devices + cmd_port + binary path + WebGUI port. VARA's transport expander is much simpler (host/cmd_port/data_port). The expander content differs per protocol — same shell pattern.
