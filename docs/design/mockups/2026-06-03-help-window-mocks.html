Tuxlink Help — separate-window mocks

Three layout candidates for the redesigned Help window. All three replace the current modal-over-client overlay with a separate Tauri webview (mimicking the existing compose_window.rs precedent), default to a larger reading size (~18–19 px), and ship an in-app font-size control. The variants differ in how the table of contents and topic navigation are presented.

bd-tuxlink-0gsy · agent bog-bluff-mesa · 2026-06-03 · content sample drawn from docs/user-guide/02-connections.md

Variant A — Sidebar ToC + reading pane closest evolution

Direct evolution of the current modal: a 260 px left sidebar lists all ten topics, the right pane renders the active one. Header carries an optional search affordance and the font-size cluster. Familiar to any operator who has used the current modal; no relearning needed.

Connections

Tuxlink reaches a Winlink Common Message Server (CMS) over one of three transports: telnet over the internet, packet radio over VHF/UHF, or ARDOP over HF. Pick the one that matches your radio, license class, and current operating conditions.

Packet (1200-baud AX.25)

Packet uses AX.25 over a TNC or a soundcard modem (Dire Wolf with KISS TCP). It carries short Winlink B2F sessions over VHF / UHF FM.

  • Bandwidth: 1200 baud — small messages, no large attachments.
  • Latency: seconds (line-of-sight VHF) to minutes (digipeated).
  • License gate: any U.S. amateur class (no Morse requirement).
  • Use cases: local emcomm, neighborhood nets, training.

The SSID picker in the dashboard ribbon attaches -N to the callsign for each AX.25 session (N7CPZ-7 is the convention for Winlink).

Strengths: Familiar; topics always visible; section groupings give structure (Getting started / Using / Config / Reference); search and font controls both prominent in header. Weaknesses: Reading pane is narrower (~720 px max); sidebar takes 22% of horizontal real estate even when the operator knows where they are.

Variant B — Top tab strip + full-width reading browser-like

Topics live as horizontal tabs across the top, mirroring how operators already navigate browser docs (MDN, Read the Docs). Reading pane is full window width with content constrained to a comfortable ~720 px reading column centered in it. Toolbar offers Back / Forward history nav (operators read non-linearly).

Font-size control study — 4 affordance candidates widget-level

Layout A is locked. The next call is what the size-changing widget itself looks like, since the userbase can't be assumed to know that Ctrl+ and Ctrl− exist. Each row below shows the widget at production size, sitting where it would in Variant A's header. All four still keep Ctrl+ / Ctrl− / Ctrl0 as power-user shortcuts.

Two 44 × 36 px buttons, percent always visible. Symbol scaled so the “A” on each button reads as a visual size hint. Universal pattern; older operators have seen this on browsers and PDF readers. Granular (each click is one step, default ~10%).

Device-target constraint — Panasonic FZ-M1

The help window (and the main client — tracked separately as a parallel bd issue) must explicitly support the Panasonic FZ-M1: a 7″ rugged tablet with 1280 × 800 WXGA at ~216 PPI, capacitive touch, common in emergency-response deployments. This constrains the help-window design in four concrete ways:

  • Default window size cannot exceed the screen. The mock above is 1180 × 740; the FZ-M1 has ~1280 × 760 effective area after OS chrome. The help window’s default size target is ~1100 × 700 — comfortable on FZ-M1 without forcing maximize, still spacious on a desktop.
  • Sidebar collapse below a width threshold. Variant A’s 260 px sidebar + ~720 px reading column + padding totals roughly 1080 px. Below that — or if the operator narrows the window for split-screen with the client — the sidebar should collapse to a hamburger button (Variant C’s drawer pattern) automatically. Breakpoint: ~960 px window width.
  • Touch targets ≥ 44 × 44 px. The FZ-M1 is a touch device. The font-control widget sizes above (44 × 36 px pills, 8 × 14 px-padded segment buttons) need to bump up so every tap target clears the touch-comfort floor. Sidebar items and topic-navigation pager buttons too.
  • High-DPI text sizing. “Normal” on the size scale is the operator’s comfortable reading size at the device they’re using. The CSS uses logical pixels, so the OS scale factor handles physical sizing — but the size-tier labels (“Normal / Large / X-Large”) should be tested on the FZ-M1 directly before finalizing the px values.

Sister bd issue covers the main client’s parallel audit (ribbon, mailbox, radio panel, compose, settings, wizard) at FZ-M1 dimensions.

Shared across all three variants

  • Default reading size 18–19 px (vs the 14 px chrome elsewhere). Older eyes are the explicit user. The font-control cluster lets each operator dial it further.
  • Font-size buttons + Ctrl+ / Ctrl- / Ctrl0 shortcuts (matches browser muscle memory). Setting is persisted in the help window's local storage so it's remembered across launches.
  • Separate Tauri webview window labeled help, lifecycle independent of main client. Single-instance: clicking Help when the window is already open focuses it rather than spawning a duplicate. tauri-plugin-window-state persists size and position (same plugin compose-window uses).
  • Theme follow. The help window inherits the main client's current theme on launch and updates live if the operator changes themes mid-session. (Open question: is there value in letting the operator override to “always light” for reading? Older users often prefer dark text on light paper for long reads.)
  • External links still go through @tauri-apps/plugin-shell::open to the system browser (unchanged from current behavior). Inter-topic .md links stay in-window.
  • Print. Not in v1 (small marginal lift, can add later as a header button that triggers the webview's print dialog).
