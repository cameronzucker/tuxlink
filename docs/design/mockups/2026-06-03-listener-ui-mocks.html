tuxlink-xehu/inde/dhbl — frontend wiring for the listener-arms gate · 2026-06-03 · plover-magnolia-salamander

What we're designing

The three transport-listener PRs (#318/319/320) shipped backend-only. The architecture doc §2.3 calls for three UI affordances per transport: listener toggle, allowed-stations editor, station-password setter (Telnet only). The existing per-transport RadioPanel pattern (one panel per {transport, intent}) is the home for these affordances — no new navigation, no popup windows.

Showing: 2 layout options for the most complex case (Telnet-P2P listener) at the top, then the chosen pattern applied to Packet + ARDOP below.

What's NOT in scope here: the global "incoming session" notification surface (separate, comes later); the listener-status indicator in the StatusBar (a one-line addition; same pattern as existing modem status); the listener forensics log viewer (also separate).

Option A — trade-offs

  • Single "Listen" header anchor; all listener stuff under one branch.
  • Collapsed by default → quiet UI when not used.
  • Visible count chip (2 callsigns · 1 IP) tells operator the list is non-empty without expanding.
  • Three triangles in a row can feel a bit busy.
  • "Listener setup" expander is mostly inert after first config — possibly hidden too deeply.

Option B — trade-offs

  • Every affordance has equal visual weight; matches architecture-doc §2.3 framing.
  • No expanders — operator sees state at a glance.
  • Splitting the panel into "Dialer half" + "Listener half" via a separator labels the two roles cleanly.
  • Long panel — most operators don't touch most of these after first setup.
  • 5 sections in the listener half can feel heavy next to the 2 in the dialer half.
  • Concurrent dial + listen state can compete visually for "primary action."
C

Recommended: Option A — extended Listen section with collapsible details

A's compactness wins for the default case; B's flatness is for operators who never collapse anything.
Why A: the dialer half is already two compact sections. Mirroring that compactness in the listener half (one anchored "Listen" header + collapsibles) keeps the panel balanced. After first-run, the operator's typical interaction is just arm/disarm + glance at the allowed-list count chip; the details only open when they want to edit. The cost (three triangles) is real but small — and the count chip means an empty allowlist is loudly visible without expansion.

Packet — what's different

  • Existing "Auto-arm at startup" checkbox stays — it's already familiar.
  • Only one expander (no IP wildcards, no password) — quieter than Telnet.
  • No "Listener setup" expander — KISS-TCP host/port lives in the existing Modem Link section above.
  • The "Allow any peer" default flipped to OFF is a real behavior change from prior builds — needs a one-time toast or onboarding note so operators don't think their listener is broken.

ARDOP — what's different

  • Honest-state banner up top so operator doesn't expect inbound to work.
  • Arm button is still wired (logs consent + persists arms record + appends forensics) — useful for verifying the UX path.
  • No password expander (ARDOP has none).

F

Open questions

Things worth confirming before we lock in.
  1. A vs B — collapsible details (A) vs always-flat sub-sections (B)?
  2. "Peer Password" vs "Station Password" on Telnet-P2P — same panel, similar names, different roles (per-peer outbound vs per-station inbound). Is the label disambiguation enough, or should we rename one (e.g. "Per-peer dial password" + "My station password")?
  3. Live count chip vs no chip on the "Allowed stations" expander summary — does 2 callsigns · 1 IP help operator confidence enough to keep, or is it visual noise?
  4. "Allow any peer" toggle default OFF on Packet diverges from WLE-parity + from prior tuxlink behaviour (the listener used to accept all). Surface as a one-time toast on first migrated launch? Or just let the empty allowlist + reject-all behaviour announce itself?
  5. ARDOP deferred-state UX — is the warn banner + muted button enough? Or should the arm button be disabled entirely with a "Requires tuxlink-95g8" tooltip?
