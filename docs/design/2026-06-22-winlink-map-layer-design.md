## Problem statement

Plot the Winlink gateways the operator is **calling or has recently called** on the shared
Leaflet theater-of-ops map, as a **toggle layer on the existing APRS map**, and animate the
**live** RF link to the connected gateway so the operator can see, at a glance, what is
happening on the air: connecting, sending, receiving, channel busy, dropped.

This is NOT a contacts/address-book map. The address book stores message recipients with no
RF position; only gateways the operator actually connects to have a position (via the station
catalog grid lookup). "Contacts" in the bd title resolves to "gateways you have a connection
relationship with."

## What makes it cool

The headline is a **live, protocol-aware RF-path animation** riding a curved arc between the
operator and the connected gateway. Direction, color, and line construction read out the link
state without a list: a packet comet flies operator→gateway while sending, gateway→operator
while receiving; a dashed handshake pulse on connect; an amber shimmer when the channel is
busy; a flash when the link drops. It turns the abstract "connected · 00:41 · 3.2 KB↑" status
line into something you can watch.

## Decisions locked (this session)

1. **What is plotted.** Gateways the operator is calling / has recently called, **if they have a
   position** (callsign → grid via the station catalog), windowed by recency **like APRS**
   (stale-dim → TTL-drop), **user-configurable** window. Operator's own pin included. Email-only
   address-book recipients are never plotted (no location). — *operator, 2026-06-22*

2. **Surface.** A **toggle layer on the existing APRS theater-of-ops map** (one engine, one
   canvas), NOT a new tab. This makes visual disambiguation the central constraint: "distinctive
   connection lines and icon sets so it doesn't get crazy." — *operator, 2026-06-22*

3. **No new chrome.** Just the map visualization. The modem tab already owns connection
   state/telemetry; this feature adds NO right-hand dock or side pane. — *operator, 2026-06-22*

4. **Animation fidelity = truthful-now (ARDOP).** v1 animates ONLY what the modem actually
   reports via the live `modem:status` event. ACK and RETRY frames are **not** exposed to the
   frontend and are therefore **excluded** from v1 (showing them would be fabricated). They become
   a later bd issue (Tier 2) if/when a backend frame-event channel is built. — *operator, 2026-06-22*

## Visual grammar (the "doesn't get crazy" contract)

Two trace types coexist on one canvas; they must never read alike:

| | APRS digipeat (existing) | Winlink link (new) |
|---|---|---|
| Geometry | straight, **segmented hop-by-hop** relay path | single **curved arc** operator↔gateway |
| Endpoints | APRS symbols / circles (●) | operator ● + gateway **diamond (◆)** |
| Motion | draw → linger → fade (one-shot) | live, continuous, direction = data direction |
| Trigger | heard digipeat path | the operator's active modem connection |

**Icon set** (inherited from the 2026-06-15 `connection-history-map-mock`, already validated
direction):
- **◆ diamond** = a Winlink gateway you've called. Green = reached, amber = last attempt failed,
  opacity fades with age (recency). The **live** peer gets a bright glowing diamond.
- **● circle** = APRS heard station (existing layer, untouched).
- **● blue** = operator position.

**Truthful-now event grammar** — every visual maps to a real `modem:status` signal. State
values are the **kebab-case wire literals** (serde `rename_all="kebab-case"`, `src/modem/types.ts`),
NOT the Rust enum casing:

| Visual | Real signal | Construction |
|---|---|---|
| Connecting / handshake | `state === 'connecting'` (and `quality`/`snDb` first arriving) | dashed cool-blue probe, hollow ring at head, outbound. **Triggered by the `connecting` state — there is NO frontend ping/pingack event** (see feasibility note). |
| Data out → | `state === 'connected-iss'` + `bytesTx` delta / `throughputBps` | bright-green comet, operator→gateway, frequency ∝ rate |
| Data in ← | `state === 'connected-irs'` + `bytesRx` delta / `throughputBps` | cyan comet, gateway→operator |
| Busy ⇄ | `arqFlags.busy` | amber dashed shimmer marching along the whole arc |
| Error | `state === 'error'` (failure drop) | amber-red flash, arc drops |
| Clean disconnect | `state === 'disconnecting' → 'idle'/'stopped'` (no failure) | quiet fade-out of the arc, NO amber flash (distinct from error) |
| Arc tint | `quality` (0..100) / `snDb` | base arc hue/intensity reflects live link quality |

`quality`/`snDb` are derived inside Rust from PING/PINGACK and surfaced as **numbers** on the 4 Hz
snapshot; the frontend never sees discrete ping events. The connecting visual therefore keys off
`state`, with quality/snDb feeding only the arc tint.

**Excluded from v1 (Tier 2, deferred):** ack ticks, retry stutter — no honest data source today.

## Reuse map (compose, don't reinvent)

- **`LeafletMap`** (`src/map/LeafletMap.tsx`) — the one shared engine. Children add layers via
  `useLeafletMap()` + `useLeafletLayerGroup()`.
- **`AprsPositionsMap`** (`src/aprs/AprsPositionsMap.tsx`) — the theater-of-ops engine this layer
  toggles onto; reuse its `L.circleMarker` + `L.featureGroup` add/remove pattern and the
  `AprsLayersPanel` checkbox for the layer toggle. **CSP pitfall:** size divIcons via width/height
  HTML attributes or CSS classes, never inline `style="..."` (production CSP blocks parsed inline
  styles — the v0.74.1 huge-sprite bug).
- **`StationFinderMap`** (`src/catalog/StationFinderMap.tsx`) + `stationModel.ts` — the gateway
  catalog and the callsign→grid join. `Station.grid` → `gridToLatLon()` gives the pin position.
- **`DigipeatPathLayer`** (`src/aprs/DigipeatPathLayer.tsx`) + `digipeatAnim.ts` — the cn84
  Canvas2D substrate (single canvas, z-index 450 above tiles / below markers, rAF lifecycle,
  segment geometry/trim). The **canvas + rAF scaffolding carries over directly**; the truthful
  packet grammar (comets/ping/busy driven by `modem:status`) is the new code.
- **Connection-detail popup** — reuse the established component (see the 2026-06-15 mock): hover/
  click a diamond → history (attempts, last connection, band/freq, quality), reusing
  `useContactConnectionRecord(callsign)` (`src/contacts/useContactConnectionRecord.ts`) — a
  **per-callsign** lookup, exactly the popup's shape. NOTE: the **diamond set** (all recently-called
  gateways within the window) is a *different* query that does NOT exist yet — build a new
  `contacts_recent_gateways(window)` command (raw data is in `favorites/store.rs`'s `log()`:
  `ConnectionAttempt.ts_local` + `.gateway`).
- **Operator position** — `useStatusData()` (`src/shell/useStatus.ts`) → `grid` →
  `gridToLatLon()` (`src/forms/position/maidenhead.ts`).
- **Live link telemetry** — `useModemStatus()` (`src/modem/useModemStatus.ts`) subscribing to the
  `modem:status` Tauri event (4 Hz). Fields: `state`, `peer`, `bytesTx`, `bytesRx`,
  `throughputBps`, `arqFlags{tx,rx,busy}`, `quality`, `snDb`, `uptimeSec`. The modem tab
  (`ArdopRadioPanel.tsx`) already consumes this — same source, new consumer.

## Feasibility / data-source map (the grounding that drove decision 4)

| Need | Status | Source |
|---|---|---|
| Gateway positions | ✅ exists | station catalog grid → `gridToLatLon` |
| Recency-windowed "recently called" **set** | ⚠️ **build a new command** | The raw data exists (`ConnectionAttempt.ts_local` + `.gateway`, all attempts via `favorites/store.rs log()`), but the only existing hook, `useContactConnectionRecord`, is **per-callsign** — it does NOT return the windowed set. Build a new `contacts_recent_gateways(window)` Tauri command/hook. |
| Per-diamond connection history (hover popup) | ✅ exists | `useContactConnectionRecord(callsign)` — per-callsign lookup, exactly the popup's shape |
| Live connection state + peer | ✅ frontend-reachable | `modem:status` event (`state`, `peer`) |
| TX/RX bytes + throughput | ✅ frontend-reachable | `bytesTx`/`bytesRx`/`throughputBps` |
| Connecting / handshake | ✅ frontend-reachable | `state === 'connecting'` (NOT a ping event — no discrete PING/PINGACK reaches the frontend) |
| Channel busy | ✅ frontend-reachable | `arqFlags.busy` |
| Link quality / SNR | ✅ frontend-reachable | `quality` / `snDb` **numbers** (PING-derived inside Rust, held across session; surfaced on the snapshot, not as events) |
| Per-frame ACK / RETRY | ❌ backend-only | TNC frame stream stays in Rust; needs a new transport-level event channel (Tier 2) |
| Per-session replay log | ❌ does not exist | only aggregate records → hover shows the popup, NOT an animated replay |
| VARA live animation | ⚠️ blocked | VARA has no `ModemTransport` impl yet (raw TCP); animation is **ARDOP-only** until VARA is integrated (separate bd issue) |

## Approaches considered

### Approach A — Own surface/tab *(rejected by operator)*
A dedicated "Winlink" map sibling to APRS/StationFinder. Clean data separation, but the operator
chose a single situational pane.

### Approach B — Toggle layer on the APRS map *(SELECTED)*
Winlink stations + the live link animation as a toggleable layer in the existing APRS map.
One unbounded situational index. Forces the icon/line disambiguation work, which is the point.

### Fidelity sub-decision
- **B1 — Truthful-now grammar (ARDOP)** *(SELECTED)*: animate only real telemetry; exclude
  ack/retry. Ships on the existing event; matches the alpha=vetted bar.
- **B2 — Build the frame-event channel first** *(rejected for v1)*: richer/fully-truthful full
  grammar, but adds Rust transport work and depends on an uncertain ardopcf cmd-socket spike.
  Deferred to Tier 2.

## Recommended approach (the build)

Toggle layer on the APRS map, truthful-now grammar, ARDOP:

1. **Gateway layer** — read connection-history records, filter to the configurable recency
   window, join callsign→catalog grid, drop positionless gateways. Render ◆ diamonds
   (reached/failed/recency-tiered) via the `AprsPositionsMap` featureGroup pattern, behind an
   `AprsLayersPanel` toggle ("Winlink links").
2. **Live link animation** — subscribe to `modem:status`; when `state` is `Connecting`/
   `Connected*` and `peer` resolves to a position, draw the curved arc on the `DigipeatPathLayer`
   canvas and run the truthful grammar (ping / directional comets / busy / error / quality tint).
   One link at a time.
3. **Hover/click a diamond** — connection-detail popup (history + last connection + band/freq +
   quality), reusing the established component and `useContactConnectionRecord`.
4. **Recency window control** — small on-map segmented control (1h / 6h / 24h / 7d), default
   matching the APRS stale/TTL convention.

## Live-arc edge cases (must-handle in v1)

- **Connected peer with no catalog position.** The live arc needs both endpoints; if `peer` does
  not resolve to a grid, **suppress the arc** and surface a non-geographic affordance ("connected
  to `<peer>` — no map position"). Do not guess a location. The diamond layer already drops
  positionless gateways; the arc follows the same rule.
- **Layer toggled OFF during a live animation.** The arc lives on the shared `DigipeatPathLayer`
  canvas (z-index 450, `pointer-events:none`), NOT inside the diamond featureGroup. The "Winlink
  links" toggle must gate **both** — tearing down the diamond featureGroup AND stopping the arc's
  rAF driver. Toggling on again re-attaches to the current `modem:status`.
- **Live peer falls outside a newly-narrowed recency window.** Keep the **live peer's diamond
  pinned visible regardless of the window** (you are connected to it right now); the window only
  governs historical gateways. The arc and its endpoint diamond never disappear mid-session.
- **Clean disconnect vs error.** `state === 'error'` (or a failed disconnect) = amber-red flash.
  A normal `disconnecting → idle/stopped` hang-up = a quiet fade-out, no flash. The two must read
  differently so a clean QSO end doesn't look like a drop.

## Prerequisite

**None of the reused map code is on `main` yet** — `LeafletMap`, `AprsPositionsMap`,
`StationFinderMap`, `DigipeatPathLayer` live on the unmerged Leaflet-migration / cn84 branches
(PR #849, epic tuxlink-u3qe). This feature **depends on those landing on the target branch first**;
sequence the bd deps accordingly (`bd dep add tuxlink-s1o1 <map-epic-id>`).

## Open questions / deferred

- **Tier 2 (new bd issue):** backend ARQ frame-event channel → real ack/retry visuals. Needs a
  spike: can ardopcf's cmd-socket cleanly yield ack/nak/retry/frame-type?
- **VARA animation (new bd issue):** depends on a VARA `ModemTransport` impl. Until then the live
  arc animates ARDOP only; VARA gateways still appear as diamonds (from history) but don't animate.
- **Recency window default value** — pick to match the existing APRS `STALE_MS`/TTL constants
  (confirm exact defaults during implementation).
- **Concurrent links** — Winlink connects one gateway at a time, so one live arc is the norm;
  no multi-arc coordination needed for v1 (the cn84 layer supports it later if ever needed).

## Success criteria (definition of done)

- Toggling "Winlink links" on the APRS map shows ◆ diamonds for recently-called, positioned
  gateways, recency-tiered, behind the existing layers panel, with the operator pin.
- During a live **ARDOP** Winlink connection, the arc to the connected gateway animates from real
  `modem:status` telemetry: handshake on connect, directional comets while TX/RX, busy shimmer,
  quality-tinted arc, error flash on drop. Nothing on screen is fabricated.
- Hover/click a diamond shows the connection-history popup.
- The Winlink layer is visually unmistakable from APRS symbols + digipeat traces on the same
  canvas (diamond vs circle; curved arc vs straight segments).
- No new window/dock; no inline-style divIcons (CSP-safe); abort/no-runaway-TX bar unaffected
  (read-only consumer of telemetry — no TX path touched).
