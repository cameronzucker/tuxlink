## 1. Goal & scope

Add AX.25 1200-baud packet as a Winlink transport in tuxlink's native client, so an
operator can move Winlink traffic over VHF/UHF packet — to an RMS Packet gateway, to
a peer station (P2P), and by **answering** an inbound peer call — using a KISS modem
reached over TCP, USB serial, or Bluetooth serial. Target: an installable, working
v0.1.

**In scope (v0.1):**
- **Modes:** dial an RMS Packet gateway · dial a peer (P2P) · **listen/answer** an inbound peer call.
- **Transports (KISS byte-pipes):** TCP (Dire Wolf / SoundModem KISS port) · USB serial · Bluetooth serial (RFCOMM, treated as a serial device).
- **AX.25:** connected-mode v2.x, **mod-8**, 1200 baud, **digipeater path 0–2 relays** (0 = direct; ≥1 required for the operator's at-home gateway reach — load-bearing).
- **SSID:** a **global, sticky, persisted** station SSID (operate as e.g. `N7CPZ-7`).
- **Frequency:** a **note-only** field (operator tunes the radio). Future rig/freq control plane: [`tuxlink-5jb`](#8-out-of-scope--deferred).

**Out of scope (deferred):** AGWPE protocol, Linux-kernel-AX.25 (`kissattach`) path, 9600 baud, mod-128, RadioOnly/PostOffice/MESH session roles, packet-gateway directory sync, rig control (`tuxlink-5jb`).

## 2. Operational modes (ground-truthed)

tuxlink is the **dialer** in two modes and the **answerer** in one. The B2F primitives
(`handshake`/`proposal`/`transfer`/`lzhuf`/`secure`/`message`) are reused unchanged; the
*orchestration role* differs.

| Mode | tuxlink role (FBB) | Who speaks first | Secure-login | Auth source |
|---|---|---|---|---|
| **Dial gateway** | slave (dialer) | gateway (master) | **yes** — answer `;PQ` with `;PR:` | keyring Winlink password |
| **Dial peer (P2P)** | slave (dialer) | peer (master) | **no** (peer never challenges) | RF callsign only |
| **Answer peer (listen)** | **master** (answerer) | **tuxlink** (master) | **no** | RF callsign only |

- The dialer cases reuse today's `run_exchange` exactly (it is slave-role: "server speaks first", conditional `;PQ` already handled).
- The answerer case needs a **new master-role path** (handshake-first; remote/slave takes the first message turn).
- **Gateway vs peer is not an operator choice** — tuxlink dials a call sign and reacts: answer a challenge if one arrives (gateway), proceed without if none (peer). One "Connect to" control.
- **Idle = listening** (default-on): when not actively transmitting, the AX.25 layer is armed to answer an inbound SABM to our call. Packet is **half-duplex**: an in-progress outbound session means an inbound call is simply not answered until the channel is free (the modem/CSMA arbitrates access; tuxlink does not implement CSMA).

## 3. Architecture (layering)

Bottom-up; three new layers below the session + one new path within it. Mirrors how the
existing client was built (message → compression → exchange → telnet).

```
⑥ UI + config        (extend: Connections-section Packet entry, reading-pane
                       connection panel, ribbon/status transport+listen, SSID,
                       session-log packet lines)
⑤ Orchestration      (extend: TransportConfig::Packet{link, role}, dial/listen
                       lifecycle, abort/timeout)
④ B2F session        (REUSE run_exchange for dial; ADD master-role path for answer)
③ AX.25 v2.x         (NEW — connected-mode data-link state machine; connect+answer;
   connected-mode      digipeater path; presents Read + Write)
② KISS framing       (NEW — FEND/FESC, command frames for TNC params)
① KissLink transport (NEW — TCP | USB-serial | BT-serial; yields a byte stream)
```

The AX.25 engine presents the same `Read + Write` interface that `telnet.rs` hands to
`session::run_exchange` — that is the seam that makes session reuse possible.

## 4. Component design

### 4.1 `KissLink` transport (`winlink/ax25/link.rs`)
A trait yielding a bidirectional byte stream + connect/reconnect; three impls:
- **TCP** (`TcpStream`): host:port to Dire Wolf/SoundModem's KISS port (default 8001). Loopback-testable with no radio.
- **USB serial** (`serialport` crate): COM device + host-link baud (distinct from over-air 1200).
- **Bluetooth serial:** **decision** — treat as an OS-exposed serial device (`/dev/rfcommN`); operator pairs + binds at the OS; tuxlink opens it identically to USB-serial. No in-app BlueZ dependency. (Matches how RMS Express sees a BT TNC: a COM port.) tuxlink's only BT-specific surface is pairing help text.

Mirror the telnet path's abort hook (`register_socket` → external `.shutdown()`/close) so a hung connect/exchange can be aborted.

### 4.2 KISS framing (`winlink/ax25/kiss.rs`)
- Encode/decode KISS frames: `FEND`(0xC0)/`FESC`(0xDB)/`TFEND`(0xDC)/`TFESC`(0xDD) escaping; type+port command nibble; data frames (cmd 0).
- Param-set command frames: `TXDELAY`(1), `P`(2), `SlotTime`(3), `TXtail`(4), `FullDuplex`(5) — applied on open from the AX.25 timing config.

### 4.3 AX.25 v2.x connected-mode engine (`winlink/ax25/`)
The correctness-critical core. Modules:
- `frame.rs` — address field (callsign + 4-bit SSID + C/R + last-address bits), control fields (SABM 0x2F · UA 0x63 · DISC · DM · I · RR/RNR/REJ · FRMR · UI), N(S)/N(R), FCS (CRC-16/X.25), command/response semantics.
- `path.rs` — source/dest + **0–2 digipeater** address list, the has-been-repeated bit handling.
- `datalink.rs` — the state machine: `Disconnected → {AwaitingUA (connect) | AwaitingSABM (answer)} → Connected → AwaitingRelease`; T1 retransmit timer, MAXFRAME window, PACLEN segmentation/reassembly, REJ/RNR handling, N2 retry cap.
- Public API:
  - `connect(target, &[digi], params) -> Result<Ax25Stream>` — send SABM (with digi path), await UA.
  - `answer(mycall_ssid, params) -> Result<(peer, Ax25Stream)>` — await an inbound SABM addressed to our call, reply UA, surface the peer call.
  - `Ax25Stream: Read + Write` — reliable in-order bytes; `Drop`/`disconnect()` sends DISC.

### 4.4 B2F session — role-parameterized (`winlink/session.rs`)
Add `enum ExchangeRole { Dial, Answer }` to `run_exchange` (or a thin `run_exchange_master`):
- `Dial` (slave): unchanged — read remote handshake, optional `;PR:` response, `my_turn = true`.
- `Answer` (master): build+send our handshake **first** (no `;PQ` — clients never challenge), read the peer's handshake, then `my_turn = false` (remote/slave proposes first).
- **Identity split (design point + verify):** the AX.25 *link* uses the SSID'd call (`N7CPZ-7`); the **B2F `;FW:`/`DE` identity uses the base callsign** (`N7CPZ`) — Winlink accounts have no SSID. `ExchangeConfig.mycall` stays the base call; the SSID lives in the link/transport config.

### 4.5 Orchestration & config (`winlink_backend.rs`, `config.rs`)
- `TransportConfig::Packet { link: KissLinkConfig, ssid: u8, role: PacketRole }` where `PacketRole = DialTo{ call, path: Vec<Call≤2> } | Listen`.
- **Endpoint/role resolution** sibling to `resolve_cms_endpoint`.
- **Lifecycle:** idle → `answer()` armed (if Listen enabled, default on) → on inbound CONNECTED, run `Answer` exchange in background → return to listening. Operator "Connect to" → `connect()` → run `Dial` exchange → return to listening.
- **Config additions:** `[packet]` — global sticky `ssid` (0–15, persisted), last KISS link (kind + host:port / device), AX.25 timing (TXdelay/PACLEN/MAXFRAME/retries/persistence/slot — 1200 defaults), `listen_default = true`.
- **Keyring:** gateway dial reuses the existing Winlink password entry; P2P/answer use no secret.

### 4.6 UI (frontend — `src/shell/`, `src/session/`, config form)
Faithful to the approved shell (Mock B). Additions only:
- **Connections section** (sidebar): add a `Packet (AX.25)` entry beside Telnet/CMS-SSL, with a transport-state dot (green = listening/connected).
- **Reading-pane connection panel** (no pop-up window): selecting the Packet connection shows Modem (TCP/USB/BT, "Change" expands in place) · **My station** (base call + **SSID** selector → effective `N7CPZ-7`) · default-on **Listen** toggle · **Connect to** [call] + **Digipeater path** (0–2 relay chips) + Connect.
- **Ribbon Connection item + StatusBar:** show active transport + listen/connected state (`Packet 1200 · Listening as N7CPZ-7`). Honors the triple-confirmed **transport-visibility** principle (explicit, operator-chosen, echoed in status).
- **SessionLog** (existing full-width bottom strip): human-shaped packet lines (inbound ◀, outbound ▶ with digi path, SABM→UA), `Show raw` reveals AX.25/KISS detail.
- **SSID setting:** global + sticky (Settings + the panel selector write the same persisted value).

## 5. Error handling
- **Connect (no UA):** bounded by N2×T1; surface "No answer from `<call>` (via `<path>`)" — never a silent hang. Distinguish from KISS-link failure (modem/TCP/serial down).
- **Link failure mid-exchange:** propagate as `ExchangeError`; log the partial outcome (sent/received so far); return to listening.
- **Channel busy:** the modem/CSMA owns it; tuxlink waits. Surface "channel busy" only if the modem reports it.
- **Inbound while busy:** not answered until free (half-duplex) — expected, logged.
- **Abort:** operator-cancellable connect/exchange via the link's shutdown hook (mirror telnet `tuxlink-9z2`).

## 6. Testing & RADIO-1 boundary
**Agent-runnable (no RF, no transmission):**
- Unit: KISS frame round-trip (escaping edge cases); AX.25 `frame.rs` encode/decode incl. SSID + digi path + FCS; `datalink.rs` driven by a scripted in-memory peer (SABM/UA, I/RR/REJ, T1 timeout/retransmit, DISC); `session` Dial **and** Answer roles over in-memory streams.
- Integration (optional, no TX): a **local Dire Wolf in software loopback** for an end-to-end KISS path with no transmitter keyed.
- **Test plan covers BOTH a direct (0-relay) connect AND a digipeated (≥1-relay) connect**, so neither path is first exercised live at release.

**Operator-only (on-air, licensee-run per RADIO-1):** real digipeated connect to the gateway; live P2P dial; live P2P **answer** with a second station. The agent builds + verifies in-memory/loopback; the operator runs every on-air step. "On-air testing will tell the full story" (operator, 2026-05-22).

## 9. Robustness pipeline & open verification items
The AX.25 connected-mode engine (§4.3) is correctness-critical and hard-to-undo → **full pipeline**: TDD against the AX.25 v2.2 spec, cross-checked against `TNCKissInterface.dll` frame types (second reference) + `wl2k-go`, plus the cross-provider **Codex** adversarial round. Lower layers (KISS framing, transport adapters, config) and the UI follow normal TDD.

**Verify during implementation (do not assume):**
1. Exact FBB turn-order for **both** roles vs the existing `run_exchange` loop (Dial vs Answer first-turn).
2. Whether the B2F `;FW:`/`DE` carry base call or SSID'd call (§4.4) — confirm against wl2k-go + RMS Express.
3. KISS `ACKMODE` vs `NORMAL` handling for the chosen modems.
4. Dire Wolf / SoundModem default KISS-TCP ports + any handshake quirks.
5. SABM vs SABME selection (mod-8 only for v0.1 → SABM).
