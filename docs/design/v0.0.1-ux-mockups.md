## 1. Scope and authority

### 1.1 What tuxlink IS — and what it is NOT

**Tuxlink is a Winlink _client_, not a Winlink _gateway_.** Specifically, tuxlink replaces (or offers a better path than) **Winlink Express** — the end-user client program for amateur radio operators to compose, send, and receive Winlink mail. Tuxlink does NOT replace **RMS Trimode** (or RMS Packet, RMS Pactor, RMS Relay) — the gateway/server software that station operators run to receive *incoming* radio connections from clients and bridge them to the Winlink CMS.

The Winlink architectural layering:

```
Client (Winlink Express, Pat, → tuxlink)
       │
       │  initiates connection over Telnet (CMS-direct) or radio (Packet/Pactor/VARA HF/VARA FM)
       ▼
Gateway (RMS Trimode, RMS Packet, RMS Pactor, RMS Relay)   ← out of tuxlink's scope at any planned version
       │
       │  bridges to internet (or holds locally in Hybrid Network mode)
       ▼
CMS (cms.winlink.org)
```

Tuxlink replaces the top box only. It does not implement gateway functionality. Operators who want to *run a Winlink gateway* — listen for incoming radio connections and bridge them to the CMS — continue to use RMS Trimode (or its successors) for the foreseeable future.

A note on the legacy file naming: Winlink Express was originally named "RMS Express" and renamed in June 2016 (per the CHM `hs10.htm`). Installation files still carry the legacy name (`RMS Express.exe`, `RMS Express.ini`, `RMS Express.chm`, etc.) for installation-folder compatibility, but the product **is** Winlink Express. This is a frequent source of confusion when reading a Winlink install directory — "RMS Express" and "Winlink Express" refer to the same client product; "RMS Trimode" (and other "RMS *" gateway products) is a different product with a different role.

Future agents who encounter a "should tuxlink also do gateway operation?" suggestion should be pointed at this section + at SCOPE-1 in [`docs/pitfalls/implementation-pitfalls.md`](../pitfalls/implementation-pitfalls.md) and told no.

## 3. Locked decisions (the five tensions from brainstorm)

These were resolved in the 2026-05-17 brainstorm session and are reflected in PR #29's [mockup gallery](mockups/README.md). Restated here as locked specs:

1. **Dashboard scope** — Compact ribbon above panes (callsign · grid · GPS · UTC+local · connection), NOT a heavy emcomm console. Heavier signals (session timer, outbox queue, last-sessions log) live in the togglable Radio Dock. See [mock-b-principles-faithful.png](mockups/images/mock-b-principles-faithful.png) for the default-state ribbon; [synthesis-dock-off.png](mockups/images/synthesis-dock-off.png) / [synthesis-dock-on.png](mockups/images/synthesis-dock-on.png) for ribbon + dock interaction.

2. **Compose: separate Tauri window** — Not Radix Dialog (the current plan Task 14 spec). All surveyed native desktop email clients (Apple Mail, Thunderbird, Spark, Mimestream, Mailspring, Airmail, Geary, Evolution) use a separate window; Express does too. Radix Dialog was the outlier. Permitted exception under [Principle 2 (single-pane preference)](v0.0.1-ux-principles.md#2-single-pane-preference-multi-window-is-a-last-resort): operators want to refer to other messages while composing.

3. **Session log: human-shaped default, raw toggle** — Same log ring projected two ways: a "Connected to Winlink CMS via telnet · 3 received · 1 sent" summary by default, with a `Show raw` toggle that reveals the underlying B2F protocol lines + `***` annotations. NOT two parallel logs. Per [Principle 6](v0.0.1-ux-principles.md#6-display-human-shaped-status-not-protocol-logs). See §4.4 for the actual Express log format the toggle exposes.

4. **Layout architecture: sidebar + 2-pane, with togglable 4th pane** — Standard 3-region desktop email layout (folders sidebar · message list · reading pane) by default. View → Show Radio Dock (Ctrl+Shift+M) adds a 4th pane on the right showing dock content. See [synthesis-dock-on.png](mockups/images/synthesis-dock-on.png).

5. **v0.5+ modem console placement: 3-state toggle (Off / Compact / Full)** — Compact in-dock mode for high-volume Mode-1 operators; full-overlay mode replacing the reading pane for marginal-conditions Mode-2 operators. v0.0.1 ships the dock with a "v0.5+ modem console placeholder" block; full modem UI lands when VARA-native arrives. See [modem-compact.png](mockups/images/modem-compact.png) and [modem-full.png](mockups/images/modem-full.png).

## 4. Findings from Express artifact audit

These are new design-influencing findings from the static audit, not previously captured in the principles or anti-patterns docs.

### 4.1 Transport visibility — triple-confirmed anti-pattern

The transport-visibility anti-pattern was [first observed](../ux-anti-patterns.md) from Cameron's runtime UI inspection: Express's session-type dropdown shows only "Telnet"; the session-settings dialog only exposes port 8772; CMS-SSL (port 8773, TLS-wrapped) appears only in the post-connection session log.

The static audit triple-confirms this is design intent, not a UI glitch:

| Source | What it shows | Verdict |
|---|---|---|
| Runtime UI (Cameron 2026-05-17) | Dropdown says "Telnet"; port field 8772 | UI hides CMS-SSL |
| CHM help `hs200.htm` (Telnet Setup) | Describes only port 8772 for RMS Relay; **no mention of CMS-SSL or port 8773 anywhere** | Docs hide CMS-SSL |
| Cameron's per-callsign session log (`<CALLSIGN>/Logs/RMS Express 20260503.log`) | Every background session logs `Connecting to CMS CMS-SSL at cms.winlink.org port 8773` and `Connected to CMS-SSL` | Binary preferentially uses CMS-SSL |
| `Winlink_Express_Revision_History.txt` grep for `SSL\|8773\|TLS` | **Zero hits across 2769 lines of release notes** | CMS-SSL was introduced silently and never documented |

**Tuxlink design response** (already specified in the anti-pattern doc; restated here for the per-task specs in §5 to cite):
- The Session-type / connection chooser must enumerate transports explicitly: "Telnet (plaintext, port 8772)" and "CMS-SSL (TLS, port 8773)" — separate entries, not a hidden auto-select.
- Default for new installs: CMS-SSL. Documented as such.
- The status bar always shows the current connection's transport.
- A connection is never silently upgraded or downgraded; switching transports is an operator action.

### 4.2 Validator rules (CHM `hs30.htm` Basic Configuration)

Express's documented validators, distilled:

| Field | Rule | Notes for tuxlink |
|---|---|---|
| **My callsign** | ≤7 chars, optional `-ssid` with values 1-15. "MARS and Amateur callsigns are checked for basic syntax." | Express applies *syntax* only; the CMS authoritatively validates on first login. Tuxlink should mirror this division of labor: loose client-side check, CMS authoritative. |
| | Callsigns with SSIDs OR >6 chars *cannot* be used in Winlink Hybrid (Radio-only) Network. | Document the constraint where it bites (when the operator switches to a Hybrid session in v0.1+). Not a v0.0.1 wizard concern. |
| **My password** | Case sensitive, **6-character minimum**. | Tuxlink: enforce 6-char min; don't impose additional complexity rules (CMS doesn't). |
| **Grid square** | 6-char required. Used in ID frames, propagation calculator, GIS forms. Manual entry overridden by connected GPS. "Approximate ±1° lat/lon usually sufficient for propagation prediction." | Tuxlink (per [Principle 7](v0.0.1-ux-principles.md#7-position-privacy-via-precision-reduction-not-opt-out)): broadcast 4-char by default, store 6-char locally, opt-in to broadcast 6-char. |
| **Tactical address** | ≤12 chars. First part: alphabetic only (no digits). Optional dash + alphanumeric suffix. Valid: `RESTSTOP-1`, `REDCROSSTN`, `REDCROSS-TN`, `TEMA`, `EMCOMM-47A`. Invalid: `THISISTOOLONG`, `RESTSTOP1`, `AA1RN`. | Tuxlink: NOT a v0.0.1 surface — multi-callsign + tactical addresses are v0.1+. But the wizard's loose callsign validator (per amendment Task 10) accepts tactical-style strings without claiming to validate them; the CMS sorts it out. |
| **Callsign Suffix** (for Radio ID only) | Used for portable / marine mobile / country-code legal ID. Does NOT alter operational callsign or Winlink address. | Tuxlink: a v0.1+ Settings field. Not in wizard. |
| **Password recovery email** | Required-ish (Express allows blank but warns). | Tuxlink: optional field in wizard; explicit "you can set this later" copy. |
| **Service codes** | `PUBLIC` for amateur radio (default). MARS / SHARES use different codes (obtained from the org administrator). | Tuxlink: hardcode `PUBLIC` for v0.0.1. Settings UI for service codes is v0.1+ (when MARS/SHARES support is in scope). |

### 4.4 Session log format (Cameron's real logs)

The log is a hybrid of raw B2F protocol lines + Express-injected `***` annotations. Representative excerpt (sanitized — `W4PHS` substituted for Cameron's actual callsign, `EM75ab` substituted for his actual grid):

```
2026/05/03 03:37:03 1.7.31.0 Successfully reported Winlink Express version number (1.7.31.0)
2026/05/03 03:37:04 1.7.31.0 *** SFI = 159 from https://services.swpc.noaa.gov/text/wwv.txt
2026/05/03 03:37:04 1.7.31.0 -----  Winlink Express version 1.7.31.0 starting  -----
2026/05/03 03:37:04 1.7.31.0 *** GPS tracking initialized
2026/05/03 03:37:04 1.7.31.0 Background
2026/05/03 03:37:04 1.7.31.0 Background *** Beginning background connection to CMS ***
2026/05/03 03:37:04 1.7.31.0 Background *** Connecting to CMS CMS-SSL at cms.winlink.org port 8773
2026/05/03 03:37:04 1.7.31.0 Background *** Connected to CMS-SSL at 2026/05/03 03:37:04
2026/05/03 03:37:05 1.7.31.0 Background [WL2K-5.0-B2FWIHJM$]
2026/05/03 03:37:05 1.7.31.0 Background ;PQ: 02166776
2026/05/03 03:37:05 1.7.31.0 Background CMS>
2026/05/03 03:37:05 1.7.31.0 Background    ;FW: W4PHS
2026/05/03 03:37:05 1.7.31.0 Background    [RMS Express-1.7.31.0-B2FHM$]
2026/05/03 03:37:05 1.7.31.0 Background    ;PR: 70336572
2026/05/03 03:37:05 1.7.31.0 Background    ; WL2K DE W4PHS (EM75ab)
2026/05/03 03:37:05 1.7.31.0 Background    FF
2026/05/03 03:37:05 1.7.31.0 Background FQ
2026/05/03 03:37:05 1.7.31.0 Background *** --- End of session with WL2K at 2026/05/03 03:37:05 ---
2026/05/03 03:37:05 1.7.31.0 Background *** Messages sent: 0.  Total bytes sent: 0,  Time: 00:00,  bytes/minute: 0
2026/05/03 03:37:05 1.7.31.0 Background *** Messages Received: 0.  Total bytes received: 0,  Total session time: 00:00,  bytes/minute: 0
2026/05/03 03:37:10 1.7.31.0 Background *** Disconnected at 2026/05/03 03:37:10
2026/05/03 03:37:11 1.7.31.0 Background *** Successfully finished background connection to CMS ***
```

Format breakdown:
- Per-line prefix: `YYYY/MM/DD HH:MM:SS VERSION TAG` where TAG is `Background` for background-monitor sessions, blank for interactive sessions
- `*** message ***` framing for Express's narrative annotations (operator-relevant)
- `[WL2K-5.0-B2FWIHJM$]` — server greeting line (B2F protocol)
- `;PQ: NNNNNNNN` — secure-login challenge from server
- `;FW: <CALLSIGN>` — forwarding identifier
- `[RMS Express-1.7.31.0-B2FHM$]` — client identifier
- `;PR: NNNNNNNN` — secure-login response (the operator's password is never sent over the wire; this is a challenge/response derivation; per CHM `hs110.htm`)
- `; WL2K DE <CALLSIGN> (<GRID>)` — operator identification with Maidenhead grid
- `FF` / `FQ` — B2F end-of-session markers
- `*** Messages sent: N. Total bytes sent: N, Time: HH:MM, bytes/minute: N` — Express's per-session summary

**Tuxlink session-log design** (Task 15 spec, locked tension #3):

**Default human-shaped view** projects this log ring to operator-relevant lines only:
```
03:37:04 UTC · Connecting to Winlink CMS via CMS-SSL (cms.winlink.org:8773)
03:37:05 UTC · Connected. Secure-login complete.
03:37:05 UTC · Session ended. 0 sent / 0 received in 1 second.
03:37:10 UTC · Disconnected cleanly.
```

**`Show raw` toggle** reveals the full log including `;PQ`, `;PR`, `[WL2K-...]`, `FF/FQ` lines — for sysadmins debugging RMS/CMS handshake failures.

Both views are projections of the same in-memory log ring (do NOT maintain two parallel streams; that's a bug pattern). The Pat HTTP client's `/api/sessions/<id>/log` endpoint (Task 5 deliverable) is the source-of-truth log feed for both projections.

### 4.5 Message types model (CHM `page_5.html` + `hs80.htm`)

Express has four message types, selected at compose time, that govern *when* the message gets transmitted:

| Type | Transmitted during | Routing |
|---|---|---|
| **Winlink Message** | CMS connection (telnet OR radio-via-RMS-to-CMS) | Backbone Amazon cloud; deliverable to any RMS or telnet client |
| **Radio-Only Message** | Radio-only / Hybrid Network session | RMS-to-RMS relay to recipient's designated Message Pickup Station (MPS) |
| **Post Office Message** | RMS Post Office session | Local DB on the connecting RMS Relay; held for pickup |
| **Peer-to-Peer Message** | P2P session | Direct radio link to another Winlink Express client; bypasses CMS+RMS |

Critical invariant from `page_5.html`: *"Messages that do not match the session type remain in the Outbox and are not transmitted."* This is a footgun the tuxlink Outbox UI must defang — a Radio-Only message stuck in the Outbox because the operator only ever opens telnet sessions is silent failure.

**Tuxlink v0.0.1 scope reduction:** v0.0.1 ships ONLY **Winlink Message** type (Telnet via CMS-SSL is the only session type in v0.0.1; per the plan). The other three types are v0.1+ when the corresponding session types ship. The compose-window message-type dropdown in v0.0.1 should be present-but-disabled showing "Winlink Message" as the only enabled option, with a tooltip explaining that Radio-Only / Post Office / Peer-to-Peer ship in v0.1+. This is a forward-compatibility signal — operators familiar with Express recognize the dropdown immediately and understand what's coming.

### 4.6 Per-callsign first-class identity (CHM `hs50.htm` + `.ini` per-callsign sections)

Express treats callsigns as first-class identities:
- The drop-down at the left end of the menu bar is the active-callsign selector
- `(Add call sign)` is an entry in the dropdown
- Each callsign has its own message folders, log directory, and properties (service codes, etc.)
- Site properties (radio type, TNC baud) are global
- The `.ini` has `[<CALLSIGN>]` sections with per-callsign settings (password recovery email, HF indices flag, shortened callsign, callsign suffix)
- Per-callsign logs live in `RMS Express/<CALLSIGN>/Logs/`

**Tuxlink v0.0.1 scope reduction:** v0.0.1 ships single-callsign-only. The architecture (config schema from Task 2, Pat HTTP client from Task 5) should NOT assume single-callsign — it should be possible to ship multi-callsign in v0.1+ without rewriting persistence. Per the plan's Task 2 (config schema), the `Callsign` field is required and singular for v0.0.1. v0.1+ amendment: promote `callsign: String` to `callsigns: Vec<CallsignConfig> { active: usize, configs: Vec<...> }`.

### 4.7 GPS auto-grid-update (Express `.ini [GPS]` + CHM `hs30.htm`)

Express has `Update grid square from GPS=True/False` as a setting under `[GPS]`. When enabled, a connected GPS overrides manually-entered grid. From Cameron's install, this defaults to `True` once GPS is configured.

**Tuxlink design** (Principle 7 + this finding):
- Settings → Privacy → GPS state has three values (per Principle 7): `Off` / `Local UI only` / `Broadcast at chosen precision (default)`
- Settings → GPS → Auto-update grid from GPS: `On (default when GPS available)` / `Off` — separate from the privacy state. With privacy "Local UI only," the grid auto-updates locally but isn't broadcast.
- Settings → Privacy → Position precision: `4-character grid (reduced, default)` / `6-character grid (high precision)`
- v0.0.1 ships the toggles + the grid display in the dashboard ribbon. The actual GPS device integration (serial port, TCP/IP NMEA, on-Pi GPS HAT) ships in v0.1+ (out of plan scope; see plan §Status).

### 4.8 ICS-309 and Forms (CHM `hs97.htm` + `page_6.html`)

Express has two emcomm-critical features tuxlink defers to v0.1+:

- **ICS-309 auto-generation** from the message DB → PDF or CSV. Triggered via `Message → Generate ICS-309 Communication Log`. The CHM specifies all the input fields (incident name, operational period, station ID, operator name, etc.). v0.0.1: not in scope. v0.1+: a single menu item; the message DB and pat session data are the inputs.
- **Forms (HTML templates with XML data attachments)** — Express's Forms feature is its major emcomm differentiator (ICS-213, RRI Radiogram, EOC reports, etc.) but also its [most-citable UX failure](../ux-anti-patterns.md#anti-patterns-observed-in-winlink-express) (browser-window-spawning, draft-discarding, etc.). v0.0.1: not in scope. v0.1+: tuxlink ships forms as in-app webviews per the anti-pattern, never as external-browser launches.

The compose window in v0.0.1 should include a *disabled* "Select Template" button with a v0.1+ tooltip. Same forward-compatibility signal as the disabled message-type dropdown (§4.5).

### 4.9 Background polling adaptive 2-32 min (CHM `hs200.htm`)

Express's background-monitoring polling time is adaptive: starts at 16 minutes, halves to a minimum of 2 min when traffic arrives, doubles to a maximum of 32 min when sessions are idle. This is a reasonable default that respects the radio channel for active stations and minimizes wakeups for idle stations.

**Tuxlink design:** v0.0.1 ships with adaptive polling identical to Express's algorithm. Settings → Connection → Background polling has values: `Off` / `Fixed (15 min)` / `Adaptive (2-32 min, default)`. Default: Adaptive.

## 5. Per-task UX specifications

Each subsection here is the **canonical UX spec** for the corresponding plan task. The current plan section retains its implementation guidance (file paths, Rust signatures, acceptance criteria) but the UX behavior is specified here. Plan amendments to surface the deltas land in separate PRs (§9).

### 5.1 Task 9 — Wizard screen 1 (welcome / connection-type)

**Current plan question:** "Do you have a Winlink account?"

**New canonical question:** "Will this installation connect to the Winlink CMS?"

**Rationale:** The original question conflates two things: account-existence (have you registered with Winlink) and connection-intent (does this installation talk to the internet-backed CMS). They're separable:
- Operator has a Winlink account AND wants CMS connection → straight to Task 10 (credentials)
- Operator has a Winlink account but is deploying offline (radio-only emcomm, ARES drill, EOC tabletop, lab work) → Task 11.5 (offline path); credentials saved for later
- Operator does NOT have a Winlink account → still reaches Task 11.5 (an unauthenticated identifier is fine for offline use); registration link inline in Task 10 for the CMS path

**Mock:** [wizard-a-welcome.png](mockups/images/wizard-a-welcome.png).

**Two choice cards:**
1. **Yes, connect to the Winlink CMS** (default selection) — proceeds to Task 10
2. **No, this is an offline / radio-only deployment** — proceeds to Task 11.5; explicit copy mentions Winlink Hybrid Network, ARES drills, EOC tabletops, lab work as legitimate operational modes

**Acceptance criteria delta from plan:**
- Add: choice persists to config as `connect_to_cms: bool` (Task 2 schema amendment, §6)
- Add: choice can be changed post-wizard via Tools → Settings → Connection
- Remove: any "you must have a Winlink account" framing — offline deployments are first-class
- Keep: single-window wizard pattern (per Principle 2)

### 5.2 Task 10 — Wizard screen 2 (credentials, CMS path)

**Mock:** [wizard-b-credentials.png](mockups/images/wizard-b-credentials.png).

**Validator change:** the current plan specifies `^[A-Z0-9]{3,6}$` as the callsign regex. Replace with: **non-empty + no internal whitespace + ≤32 chars + ASCII-printable**. Rationale: Express itself (per §4.2) does only "basic syntax check"; the CMS is authoritative. Tactical-style strings like `EOC-1` or `BAOFENG-FM-01` are legitimate identifiers some operators use, and the strict-regex rejection blocks them at the wizard before they can attempt a real CMS validation.

**Register link inline at top:** "Don't have a Winlink account? [Register at winlink.org](https://www.winlink.org/user/register)" as a small link in the screen header, NOT a separate sub-step. The strong default (CMS path) deserves an inline solution for the missing-account case, not a wizard branch.

**"Save credentials and skip verification" button:** the operator can populate the credentials now and defer the test-send to later. Useful when the operator is configuring before deploying to a location with internet. Action: save to keychain, mark wizard complete, drop into main UI showing "Test send pending — Session → Test send to run now."

**Acceptance criteria delta from plan:**
- Replace: callsign regex `^[A-Z0-9]{3,6}$` → loose validator (above)
- Replace: callsign field label "Callsign" → "Your Winlink callsign" (consistency with Express's "My callsign")
- Add: "Save credentials and skip verification" button alongside the existing "Continue" button
- Add: Register link in header
- Keep: password input (≥6 chars per §4.2), stored to OS keyring directly in v0.0.1 (Linux secret-service / macOS Keychain / Windows CredentialManager via the Rust `keyring` crate, under `(service="tuxlink-pat", account=<callsign>)` per the cred-handling spec). The "v0.1+ promotion" framing of earlier drafts is superseded by AMD-13 of the v0.0.1 plan + the cred-handling refactor (tuxlink-mib, shipped 2026-05-18). The "pat config" persistence path no longer exists (`cfg.SecureLoginPassword` was deleted in the fork-side patch).
- Keep: optional password-recovery email field

### 5.3 Task 11 — Wizard screen 3 (verify / test-send, 4 substates, non-blocking)

## 6. Schema additions

These extend `docs/plans/2026-04-22-tuxlink-v0.0.1-plan.md` Task 2 (config schema). Land as Task 2 amendment commits per §9.

Add to the top-level config struct:
```rust
pub struct ConnectConfig {
    /// Whether this installation connects to the Winlink CMS.
    /// Set by Task 9 (welcome wizard). False = offline-only deployment.
    pub connect_to_cms: bool,

    /// Operator-chosen transport for CMS connections.
    /// CmsSsl is the v0.0.1 default; Telnet (plaintext) is available for
    /// networks that block port 8773. Per the transport-visibility
    /// anti-pattern, this is always explicit, never auto-selected.
    pub transport: CmsTransport,
}

pub enum CmsTransport {
    CmsSsl,   // port 8773, TLS-wrapped — v0.0.1 default
    Telnet,   // port 8772, plaintext
}

pub struct IdentityConfig {
    /// Required when connect_to_cms = true; matches Express's "callsign" semantics.
    /// Loose validator (non-empty, no internal whitespace, <=32 chars, ASCII printable).
    /// CMS is authoritative.
    pub callsign: Option<String>,

    /// Optional free-form station identifier for offline-mode operators.
    /// Used when connect_to_cms = false. Examples: "EOC-1", "BAOFENG-FM-01", "LabBench-3".
    pub identifier: Option<String>,

    /// 6-character Maidenhead grid (stored locally at full precision).
    /// Broadcast precision is governed by PrivacyConfig.position_precision.
    pub grid: Option<String>,
}

pub struct PrivacyConfig {
    /// Per Principle 7 (position privacy via precision reduction, not opt-out).
    pub gps_state: GpsState,
    pub position_precision: PositionPrecision,
}

pub enum GpsState {
    Off,                 // no GPS read at all
    LocalUiOnly,         // GPS used locally; never broadcast
    BroadcastAtPrecision, // default
}

pub enum PositionPrecision {
    FourCharGrid,   // EM75 — default broadcast precision
    SixCharGrid,    // EM75xx — opt-in high precision
}
```

The existing plan Task 2 specifies a flat `Callsign: String` field; that becomes `identity.callsign: Option<String>` in this amendment, with backward-compatibility migration handled by the config deserializer (treat the flat field as the new nested field for old configs).
