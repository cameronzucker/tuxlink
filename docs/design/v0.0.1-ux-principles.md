## Core stance

**Tuxlink is a desktop email client that happens to speak Winlink. It is not a Pat front-end and it is not a modernized Winlink Express clone.**

The reference user experience is closer to Thunderbird, Apple Mail, and Gmail than it is to anything in the amateur-radio software ecosystem. A new Winlink operator opening tuxlink for the first time should feel like they opened a regular email client — recognize the inbox, recognize the compose button, recognize where their account info lives. The radio-specific information is layered onto that familiar shape, prominently displayed where it matters, never hidden behind menus the way Express buries it.

Pat's web UI is a Pat-internal tool, not a model for what tuxlink should look like. The design doc explicitly says Pat is "a library with a web UI and CLI, not a desktop mail app." Looking at Pat's UI for inspiration on tuxlink's UX is the wrong direction. Look at Thunderbird and Apple Mail.

## Principles

### 1. Intuitive like Thunderbird, not technical like Pat

The default mental model is "this is an email app." Inbox, sent, compose, read — all in shapes the user already knows. Winlink-specific details (callsign, grid square, connection method, session log) are added *to* that shape, not substituted *for* it.

A new amateur radio operator who has never touched Winlink should be able to send their first message without reading documentation. An experienced Winlink Express user should land in tuxlink and not feel disoriented even though everything looks different — because the underlying mental model (inbox, sessions, sending) is the same.

### 2. Single-pane preference; multi-window is a last resort

Legacy Winlink (Express in particular) spawns a new window for almost every task: the session establishment dialog, the connection method picker, the station picker, the compose window, the form picker, the message view, the address book. The result is a desktop full of stacked windows where the user loses track of state and context.

Tuxlink prefers to keep everything in a single primary window. Operations that change context — composing a message, opening an address book, viewing an attachment — should happen in panes, drawers, or modals within that primary window unless there is a strong reason to spawn a separate window. The compose window in v0.0.1 is allowed to be its own window (because users want to refer to other messages while composing); most other operations should not be.

Menus are fine. Gmail has menus. The principle is "stop spawning windows," not "stop having menus."

### 3. Promote the dashboard information legacy Winlink hides

Information that the radio operator needs at a glance — and that Winlink Express buries in title bars or status bars or behind menus — should live in a prominent dashboard area:

- **Maidenhead grid square** — your operating location identifier. Critical to a Winlink session; legacy Winlink hides it in a settings dialog.
- **GPS status / manual position set** — whether tuxlink knows where you are, and how (auto from GPSd, or manually entered). Should be visible without a click.
- **Local + UTC time** — Winlink session timing matters, and operators routinely need both. Express shows neither prominently.
- **Callsign** — already-prominent in some Winlink clients but worth keeping prominent here.

The proportions of each are workshoppable in the brainstorm session. The principle is "promote, don't bury."

### 4. Persistent radio-connection-state pane

Take the cue from emcomm tools (Winlink-RMS-style ops dashboards, RMS Express in net-control mode, etc.): the operator wants to see the radio connection state continuously, not only during an active session.

In legacy Winlink Express, connection state is shown only after a session starts — the rest of the time the user has to remember which connection method they're configured for, which station they're targeting, etc. That's a known pain point.

Tuxlink should have a permanent or semi-permanent pane (or dashboard area) showing:
- Current connection method (telnet / packet / VARA / etc.) — when relevant
- Last session result and timestamp
- Current radio state if the radio is being managed by tuxlink (TX/RX, frequency if relevant)
- Activity indicator when a session is in progress

Exact layout is workshoppable. The principle is that the user always knows what tuxlink thinks about the radio, not just during the 30 seconds of a connection attempt.

### 5. Don't waste screen real estate

Legacy Winlink Express has a Windows-XP-era density profile: lots of toolbar with small icons, small grids, lots of empty space, important info in tiny labels. Modern desktop email clients use the available real estate to show *more* information at usable density — preview text in inbox rows, threaded conversations, sidebar with labels and folders, etc.

Tuxlink should follow the modern pattern. Inbox rows should show enough to scan a message without opening it (subject, from, date, preview snippet, attachment indicator if any). The session log should be visible (per the design doc's "Connected to Winlink CMS via telnet. 3 messages received. 1 message sent." spec) but in a dedicated pane that doesn't crowd the inbox.

### 6. Display human-shaped status, not protocol logs

The design doc is explicit: the session log shows "Connected to Winlink CMS via telnet. 3 messages received. 1 message sent." — NOT `pat: [INFO] B2F/BAP: ...`. Operators who want the raw protocol log can have a "show technical log" toggle, but the default surface is human-shaped status.

This applies more broadly than the session log: error messages should explain what to do next, progress should show what step is happening (not just a spinner), and connection failures should explain the likely cause (radio not connected, station busy, password rejected) instead of dumping a stack trace.

### 7. Position privacy via precision reduction, not opt-out

GPS is **on by default** when available. What's **transmitted** is **precision-reduced by default** — not a binary share-or-don't toggle, but with a **fully-off escape hatch** for operators who want zero GPS involvement.

The right privacy primitive for position data is "how precise is what I broadcast," not "do I broadcast at all." This is the convention amateur radio has used for decades via APRS: real Winlink gateways on aprs.fi show "Position ambiguous: Precision reduced at transmitter by 2 digits, position resolution approximately 1.2 miles" as the default broadcast precision.

For tuxlink:

- **Local UI** (dashboard ribbon, status bar, in-app display): always shows full GPS precision when available. The operator sees their real location in their own UI.
- **Broadcast / transmitted position** (in message headers, position reports, form auto-fill, P2P station info): **reduced precision by default.** For Maidenhead grid, broadcast 4-character (`EM75`) by default; user opts in to 6-character (`EM75xx`) high precision. For lat/lon, follow APRS convention of marking reduced precision in protocol headers.
- **The setting**: Settings → Privacy → Position precision · default `4-character grid (reduced)` · explicit opt-in for `6-character grid (high precision)`.
- **Fully off**: Settings → Privacy → GPS state has three values: `Off (no GPS read at all)` / `Local UI only (don't broadcast)` / `Broadcast at chosen precision (default)`. Operators who want zero GPS involvement — including not reading the local GPS device — can fully disable.
- **NOT**: a single binary "share GPS / don't share GPS" toggle. That's too crude and Pat's `gpsd.enable_http=false` pattern is too blunt for a position-aware emcomm tool.

This principle pairs with RADIO-2 (encryption-decision operator-gate in `docs/pitfalls/implementation-pitfalls.md`): both treat amateur-radio-protocol conventions as the authoritative reference for privacy and security defaults, not generic web-app defaults.

## What is NOT a UX principle

- "Look like Pat." Don't. Pat is a backend dependency; its UI is not the model.
- "Look like Winlink Express." Don't. Express is the prior art being improved upon, not the template.
- "Use every modern UX trend." Don't. The user is an amateur radio operator first, sometimes a power user, occasionally an emcomm volunteer in a stressful situation. Trendy UX patterns that hide actions behind gestures or rely on muscle memory from consumer apps fail those use cases.
