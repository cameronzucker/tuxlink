## Gating Question — RESOLVED (YES), from AREDN firmware source

**Does `sysinfo.json?services=1` still return advertised Post Office services on a
Babel-only AREDN node?** YES. Verified by reading `aredn/aredn` @ d3f7282
(committed 2026-06-10), not by inference:

- `files/www/cgi-bin/sysinfo.json` → HTTP **307 redirect** to `/a/sysinfo?<query>`.
- `files/app/main/sysinfo.ut` `services=1` handler reads **`/var/run/arednlink/services/`**
  (one file per node IP; each line `link|protocol|name`) → records
  `{name, ip, link, protocol}` where `link` is set to `""` when the advertised URL
  has port `:0/` (info-only, non-connectable).
- `/var/run/arednlink/services/` is populated by **`arednlink`**, AREDN's pub/sub
  service-propagation daemon (`arednlink-update` uploads to `/var/run/arednlink.sock`;
  `babel.uc` defines `LINK.path = /var/run/arednlink.sock`). This is **Babel-coupled,
  not OLSR**. OLSR footprint in current firmware: 1 vestigial ref (deletes old
  `olsrd_gw`). Babel: 28 files.

Conclusion: WLE's discovery broke during early Babel cutover (Orv W6BI's 2024
thread), then AREDN re-exposed service discovery via arednlink. The interface
`sysinfo.json?services=1` is durable on current AREDN. Discovery is buildable.

## Problem Statement

tuxlink can connect to an RMS Relay "Post Office" over an AREDN mesh
(`TelnetPostOfficeRadioPanel`, tuxlink-6c9y: `'local'` = LOCAL-pool login as
`<base>-L`, `'network'` = full-callsign normal Winlink routing, default
`127.0.0.1:8772` + a saved-relay favorites list). But the operator must **know
and hand-type the relay's host**. There is no way to discover which Post Office /
RMS Relay nodes are reachable on the mesh.

Real operator workflow this forces (Jan KD7ZWV, AREDN dev discussion 2026-06-10):
keep a private RMS-node list and **try them in turn until one is up**, hunting the
services page by hand because names vary. Orv W6BI (who worked directly with Phil
W4PHS, the WLE coder, through the Babel transition) confirms the use case is real
and used.

## What Makes This Cool

Kills two distinct pains in one move: **finding** the relays (no services-page
archaeology / name-guessing) and **knowing which are alive** (no dial-and-pray).
On a mesh — often the only path when internet + CMS are down — that's the
difference between "mail goes out" and "operator fumbles during an incident."

## Constraints

- Alpha bar = vettedness: ship the **complete** feature (finding + liveness), not a
  half-slice (`feedback_alpha_is_vettedness_not_built_ness`).
- RF/Part-97: discovery is IP-over-mesh (HTTP + TCP probe); initiates no
  transmission. RADIO-1 governs the *connect* click, unchanged.
- Credentials → OS keyring only, never plaintext (`feedback_no_disk_creds_default`).
- Inline UI, reuse existing panel chrome — no new windows
  (`feedback_inline_ui_no_window_clutter`).

## Premises (agreed in session)

- **P1.** Connecting to Network/Local Post Office relays already shipped
  (tuxlink-6c9y). This issue adds only *discovery*, feeding the existing connect +
  favorites flow. Not re-litigating whether the Post Office family belongs in
  tuxlink — it's in and partly built. (tuxlink running *as* a relay is separate
  and untouched.)
- **P2. LOCAL read, no network-wide traffic — stated network-citizenship principle.**
  Discovery itself is a single LOCAL GET to your own node's already-populated
  arednlink services list (zero mesh traffic). tuxlink will NOT crawl the
  supernet. Rationale (operator 2026-06-10): supernode-wide search externalizes one
  station's convenience onto an already-strained shared resource (overloaded /
  mis-cross-linked supernodes); amplifying traffic there is a bad trade. The
  optional reachability probe is the only wire traffic — bounded, on-demand, to
  advertised PO nodes only. **Do not add active supernode search later.**
- **P3.** Fix WLE's latent bugs, don't inherit them: (a) honor the configured
  master-node host as the sysinfo base URL (WLE stores "Mesh Master Node" but
  ignores it, hardcoding `localnode.local.mesh`); (b) node credentials → OS
  keyring, never plaintext `.dat`.
- **P4.** Off-mesh = graceful. When the node URL doesn't resolve, show a clear "not
  on an AREDN mesh" state — no hang, no scary error. Discovery is **additive**;
  manual host entry works exactly as today.
- **P5.** Job-to-be-done: surface advertised Post Office services under a
  recognizable label into the existing connect list, AND show which are reachable —
  killing the manual try-in-turn loop. (The corpus documents operators hitting
  nodes "found by WLE but not answering" — KF7RSF, Bill K0DZ — which is precisely
  the liveness gap.)

## Approaches Considered

### Approach A: List what the mesh advertises (minimal viable)
One-shot fetch + classify + list into the connect/favorites flow. Manual Refresh.
Effort S · Risk Low. Solves "finding" only — operator still tries in turn to find a
live one. Completeness 6/10 — rejected as a half-feature.

### Approach B: List + live reachability, as a "Mesh" tab (RECOMMENDED)
A plus a bounded liveness probe, surfaced in the existing FavoritesTabs chrome. See
"Recommended Approach." Effort M · Risk Med (probing is good-citizen). Completeness 9/10.

### Approach C: Zero-touch default target (lateral)
On entering Network PO mode while on-mesh, auto-select the nearest reachable relay
as default dial target. Effort M. Deferred — layer onto B once B's ranking data is
field-proven. Not filed.

## Recommended Approach — B

**Backend (`src-tauri`, new mesh-discovery module):**
1. `mesh_discover_post_offices()` Tauri command:
   - Resolve sysinfo base from the **configured master-node host** (P3a; default
     `localnode.local.mesh`). `GET http://<host>:8080/cgi-bin/sysinfo.json?services=1`,
     **following the 307 redirect** to `/a/sysinfo?services=1` (or request that
     path directly). NOTE: a client that does not follow 307 sees an empty result —
     this is a likely cause of field "discovery stopped working" reports.
   - Parse the JSON `services[]`: each `{name, ip, link, protocol}`.
   - **Post Office classifier (grounded):** keep services whose `name` contains
     `WINLINK` or `POST OFFICE` (case-insensitive) — Phil W4PHS's default Relay
     filter — AND `link != ""` (info-only `:0/` advertisements are non-connectable).
   - **Dial target = the record's `ip`** (numeric node IP) + the port from `link`
     (default B2F `8772`). Using `ip` avoids the documented `.local.mesh`/name-cruft
     connection failures.
   - **Bounded liveness probe:** TCP connect to each candidate `ip:port`, short
     timeout (≈1.5s), low concurrency (≈4). Record reachable + RTT. Label
     "reachable" not "verified" — TCP-open ≠ healthy B2F
     (`feedback_no_defensive_self_assertion`).
   - Rank: reachable first, then RTT. (No route-metric needed; RTT is the available
     signal. AREDN does not expose an ETX field in this record.)
   - Return DTO `[{callsign?, name, ip, port, link, reachable, rttMs?}]`.
   - **No caching in v1** — each Refresh re-fetches + re-probes (results are
     liveness-sensitive; staleness would defeat the point).
2. **Network-citizenship guardrails (code + comment carrying the P2 rationale):**
   the sysinfo fetch is a single local GET (no mesh traffic); probing only on
   explicit Refresh / panel-open, never on a timer, bounded concurrency + timeout,
   to advertised PO nodes only. A comment must state WHY, so the boundary survives
   future "add background polling / supernode search" PRs.
3. **Master-node-host setting** (config field + Settings UI) is **in scope for this
   work** if not already present — P3a depends on it. Verify against current
   `config.rs`; add if missing.

**Frontend (`src/favorites/`, `src/radio/modes/`):**
4. Add a **"Mesh"** tab to `FavoritesTabs`. (FavoritesTabs today, per source: shows
   Favorites | Recent tabs for ardop-hf/packet/vara-*; telnet shows Manual-only.
   The Mesh tab is NEW and appears only for the Network Post Office mode.) Rows reuse
   `FavoriteRow` + the existing filter-when->8-rows behavior (oi1g/PR #574).
5. Each row: reachability indicator (● reachable / ○ down / spinner probing),
   `name` (trim trailing cruft like `" - "`), and RTT as secondary text. If
   `callsign` can't be derived from `name`, render `name` alone. Same connect
   affordance as other tabs.
6. Discovered-but-unpinned nodes are **ephemeral** (`source:'mesh-discovery'`), not
   persisted. A pin/star promotes one to a saved favorite via `favorite_upsert`.
7. **States (all distinct):** (a) off-mesh → "Not on an AREDN mesh — discovery
   unavailable; enter a relay host manually"; (b) on-mesh, fetch failed/timeout →
   "Couldn't reach the mesh node — Retry"; (c) on-mesh, zero Post Offices advertised
   → "No Post Offices advertised on this mesh." Manual entry unchanged in all cases.

**Wireframe (Mesh tab in the Network Post Office panel):**
```
┌ Network Post Office ─────────────────────────────────┐
│ [ Favorites ] [ Recent ] [ Mesh ] [ Manual ]         │  ← Mesh tab is NEW
│ ── Mesh ──────────────────────────── ⟳ Refresh ───── │
│  ● W7ABC-10 Winlink Post Office  10.5.3.2:8772  12ms ★│
│  ● KD7ZWV Winlink Gateway        10.5.7.9:8772  31ms ★│
│  ○ N7XYZ Post Office             10.5.9.1:8772  down ★│
│  (filter box appears when > 8 rows)                   │
│         [ Send / Receive ]                            │
└───────────────────────────────────────────────────────┘
  ● reachable (TCP-open, not verified) · ○ down · ★ pin to favorites
```

## Open Questions (for esy7 + implementation)

- Confirm the `link` URL's host/port parsing across real advertisements (some carry
  hostnames, some IPs; some `http:`, some `winlink:` scheme). esy7 should capture a
  few real `/a/sysinfo?services=1` payloads.
- Is a bare TCP connect to the relay port an acceptable liveness signal, or does a
  dead-but-listening process produce false "reachable"? (Accept TCP-open as v1
  signal, labeled honestly; protocol-level check is a possible follow-up.)
- Master-node-host: confirm whether `config.rs` already has the field (from the
  Network PO work) or it must be added here.

## Success Criteria

- On an AREDN mesh, the Network Post Office panel's Mesh tab lists Post Office
  services from `arednlink` (via the 307-followed sysinfo fetch), filtered by the
  WINLINK/POST OFFICE name rule and `link != ""`, with live reachability + RTT.
- Connects by numeric `ip` (not node name), avoiding the `.local.mesh` cruft trap.
- Honors a non-default configured master-node host.
- Off-mesh / fetch-failed / empty states each render correctly; manual entry works.
- Discovery generates no mesh-wide traffic; probing is on-demand + bounded.
- One-action pin of a discovered relay to saved favorites.
