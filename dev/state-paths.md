# Gitignored-but-stateful paths

> **Purpose.** Enumerate paths that are `.gitignore`d but contain state the
> running project depends on. A fresh `git clone` is *not* a fresh-working
> state until these are populated. This document tells you which paths matter
> and how to restore them.

**Authority:** [`standing-conventions-cross-project.md` §11](https://github.com/cameronzucker/cz-agent-skills/blob/main/docs/standing-conventions-cross-project.md).
**Maintained by:** anyone who adds a new `.gitignore`d-but-stateful path. Update this document in the same PR that introduces the path.

---

## In-repo paths

| Path | What's in it | Source of truth | How to restore on a fresh clone |
|---|---|---|---|
| `node_modules/` | pnpm-managed JavaScript dependencies | `package.json` + `pnpm-lock.yaml` (both tracked) | `pnpm install` |
| `src-tauri/target/` | Cargo build cache (debug + release artifacts, compiled deps) | `Cargo.toml` + `Cargo.lock` (both tracked) + source files | First `cargo build` or `pnpm tauri build`; rebuilds incrementally thereafter |
| `src-tauri/gen/` | Tauri-generated capability schemas + platform-specific glue | Tauri's `tauri.conf.json` + capability files (tracked) | Auto-regenerated by `pnpm tauri dev` / `pnpm tauri build`; no manual step |
| `dist/` | Vite frontend build output | `src/` + `index.html` + `vite.config.ts` (all tracked) | `pnpm build` or `pnpm tauri build` |
| `.beads/embeddeddolt/`, `.beads/dolt/` | bd's local Dolt database — the canonical issue-tracker state mirror for this checkout | Tracked `.beads/issues.jsonl` (auto-exported on every bd write) | `bd init` (the Dolt store rebuilds from `issues.jsonl` on next `bd` command); if Dolt remote sync is later configured, `bd dolt pull` instead |
| `.beads/daemon.*`, `.beads/bd.sock*`, `.beads/last-touched`, `.beads/interactions.jsonl`, `.beads/push-state.json` | bd daemon runtime files (lock, socket, IPC log, per-machine push cursor) | Regenerated by bd at runtime; no canonical source needed | None — bd manages these on demand |
| `.claude/scheduled_tasks.lock` | Claude Code harness state from the `ScheduleWakeup` tool | Regenerated by the harness; ephemeral | None — safe to delete; will be recreated if needed |
| `.claude/skills/gstack/` | Symlinked gstack skill — points to `~/.claude/skills/gstack/` (a separate git repo) | The gstack repository at `https://github.com/garrytan/gstack` | `git clone` gstack into `~/.claude/skills/`, then ensure the symlink exists (or use whatever bootstrap script the user maintains) |
| `.claude/worktrees/` | (Forthcoming, D-section.) Worktrees keyed by `bd-<id>/<slug>` once the worktrees-mandatory workflow lands | Branches + bd issues; each worktree's purpose is auditable via `bd show <id>` | `git worktree add` per ADR D2's worktree-issue ownership rule (once D2 lands); not used pre-D-section |
| `.env` | Local secrets (Codex auth, any API keys this project needs) | Operator-maintained out-of-band | Operator-provided per machine; never tracked |

## Out-of-repo paths the project cares about

These live outside the repo working tree but are part of tuxlink's runtime contract. Documented here so a fresh-clone restore-from-zero knows about them.

| Path (XDG defaults) | What's in it | Source of truth | How to restore on a fresh clone |
|---|---|---|---|
| `$XDG_CONFIG_HOME/tuxlink/config.json` (defaults to `~/.config/tuxlink/config.json`) | Tuxlink's user config (callsign, grid, pat_mbo_address, etc.) per the schema in `src-tauri/src/config.rs` | Written by the wizard flow (Task 9-11, not yet implemented) | Run the wizard; or hand-author per the `Config` schema for dev/testing |
| `$XDG_CONFIG_HOME/pat/config.json` (defaults to `~/.config/pat/config.json`) | Pat's own config (callsign, password, locator) — Pat's format, NOT tuxlink's | Written by the wizard's Pat-setup step (later) | Hand-author with `mycall`, `secure_login_password`, `locator` for dev; or use Pat's own `pat init` |
| `$XDG_RUNTIME_DIR/tuxlink/pat.pid` (falls back to `/tmp/tuxlink-$USER/pat.pid` if XDG unset) | Tuxlink's PID file for the spawned Pat child process | Written by `PatProcess::spawn()` on startup; removed by `PatProcess::shutdown()` | None — managed by tuxlink at runtime |
| `~/.cargo/`, `~/.rustup/` | Rust toolchain (cargo, rustc, rustup) | rustup's installer at `https://sh.rustup.rs` | `curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs \| sh -s -- -y --default-toolchain stable --profile minimal` |
| `/usr/bin/pat` | Pat binary (la5nta/pat v1.0.0) | la5nta/pat GitHub releases | `curl -sSL -o /tmp/pat.deb https://github.com/la5nta/pat/releases/download/v1.0.0/pat_1.0.0_linux_arm64.deb && sudo dpkg -i /tmp/pat.deb` (arm64 for the Pi; swap arch as needed) |

## How to add an entry

When you add a new `.gitignore` entry that contains state worth preserving (or worth knowing how to rebuild), add a row to this document in the same PR. If the state is purely ephemeral noise that can be deleted with no consequence (logs, editor cruft), no entry needed.

The bar: "if a new contributor cloned the repo on a clean machine, would they wonder where this state comes from, OR would the missing state silently break something?" If yes to either — add a row.
