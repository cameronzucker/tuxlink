# Winlink Programs Group — research corpus, 2026-06-04

This directory is a **durable, reproducible research artifact** capturing
4,105 thread records from the public Winlink Programs Google Group
(https://groups.google.com/g/winlink-programs-group) along with the
scripts that produced and processed them. The corpus and its derived
theme quantifications are the evidence base behind the synthesis at
[`../2026-06-04-winlink-group-pain-points.md`](../2026-06-04-winlink-group-pain-points.md).

The artifact is preserved here so that future arguments — about
tuxlink's positioning relative to the Winlink ecosystem, about specific
recurring user-pain themes, about the case for any architectural or
process changes at the Winlink layer — can be made with **hard evidence
rather than operator anecdotes**, and so that any reader (Winlink team,
journalists, advocates, community members, the operator's future self)
can independently verify the claims by re-running the analysis against
the committed data.

---

## Files in this directory

| File | Purpose |
|---|---|
| `corpus.jsonl` | The 4,105-record redacted corpus, one JSON object per thread. Author callsigns + first names preserved (FCC-public); all detected PII redacted. |
| `themes.tsv` | The theme-quantification table — 23 patterns scored against the corpus. Regenerated by `themes.py`. |
| `scrape.py` | The Playwright-based scraper used to collect the corpus, exactly as it ran. Reproducibility document. |
| `redact.py` | The PII redactor applied to the raw corpus to produce `corpus.jsonl`. |
| `themes.py` | The theme-quantification script. Reads `corpus.jsonl`, writes `themes.tsv`. |

## How the corpus was collected

**Date:** 2026-06-04 04:11 PT through 14:00 PT (the run was killed at
14:00 PT when Google's automation detection began silently degrading
thread URLs).

**Method:** Playwright Chromium driven by [`scrape.py`](scrape.py),
authenticated with cookies exported from the operator's Firefox
session via the Cookie-Editor extension. The operator is a member of
the Winlink Programs Group and has legitimate read access to the same
content via their browser; the scrape automated the same read access
the operator had manually. Cookies were exported, used for the scrape,
and deleted at the end of the session — no credentials persisted
on-disk beyond the run.

**Pagination:** Google Groups serves the listing in pages of ~30
distinct thread URLs (each thread appearing as multiple anchors on the
listing page; the scraper deduplicates by URL). The scraper walked the
listing via the `[aria-label="Next page"]` button across 200 pages
collecting 6,002 unique thread URLs.

**Per-thread extraction:** For each URL, the scraper navigated to the
thread page, waited for DOM ready, and extracted:
- Subject (the `<h1>` element)
- OP author (the first `<h3>` element — typically the operator's
  display name and callsign)
- Post bodies (all `div[role="region"]` elements, one per post, capped
  at 6 KB body per post and 50 posts per thread)
- Reply count (the count of post containers minus 1)
- Scrape timestamp (UTC)

**Politeness pacing:** ~4-5.5 seconds between thread visits with random
jitter to avoid aggressive automation patterns. The full run took
about 10 hours.

**The bot-flag boundary:** At approximately thread 4,112, Google's
anti-automation system began silently degrading thread URLs: GETting
any thread URL after that point returned the group's landing page
content (h1="Winlink Programs Group", 0 message regions, 0 posts)
instead of the actual thread. The same URLs continue to render
correctly when visited manually in a logged-in browser. The scrape
was stopped, checkpointed records preserved, and degraded records
filtered out. **88.7% of records (4,105 of 4,626) were successfully
scraped before the flag triggered.**

This is a known and accepted limitation of the methodology. Going
deeper would require either IP rotation or waiting out Google's flag
and retrying with fresh cookies — neither cheap, and the operator's
read was that 4,105 threads was more than enough material for the
intended analysis.

## What was redacted, and why

[`redact.py`](redact.py) was applied to every post body before the
corpus landed in this directory. The redactor strips:

| Category | Sentinel | Rationale |
|---|---|---|
| Email addresses (including Google's partial-redaction form like `KC1...@domain.tld`) | `[email-redacted]` | Personal/identifying; redistribution without consent is a privacy concern. |
| Phone numbers (NA-style with optional country code and extensions) | `[phone-redacted]` | Personal contact info. |
| Street addresses (heuristic: number + words + Street/Ave/Hwy/etc., with optional apartment/suite suffix) | `[address-redacted]` | **Home addresses found in 24 raw threads** — exact street numbers + city + ZIP. Doxxing risk. |
| City + state + ZIP (typical follow-on to street address) | `[city-state-zip-redacted]` | Same rationale; redacting the full address. |
| IPv4 addresses (standalone, not version strings like `v1.5.28.2`) | `[ip-redacted]` | Possible internal network exposure. |
| URLs with auth/tracking tokens | `[tracking-url-redacted]` | Possible auth leak. |
| SSN-shaped numbers | `[ssn-redacted]` | Identifier of last resort. |

**What was deliberately KEPT:**

- **Callsigns** (e.g., `W4PHS`, `N5TW`) — these are FCC-publicly-registered
  identifiers. The whole point of amateur radio identification is that
  callsigns are public; the Winlink ecosystem is keyed off them.
  Redacting callsigns would defeat the corpus's purpose as evidence
  ("operators experiencing X" needs to identify operators by their
  callsigns to be meaningful).
- **First names paired with callsigns** — pairing is already public via
  FCC ULS lookup; preserving it doesn't add identifiability beyond
  what's already public.
- **Subject lines** (user-authored, posted to a forum the posters knew
  was archived).
- **Technical content** — software versions, hardware model strings,
  error messages, command syntax, etc. None of this is PII.
- **Domain names** (`gmail.com`, `winlink.org`, etc.) — generic.

**Redactor application counts (from `redact.py` output on this run):**

| Category | Hits redacted |
|---|---|
| Email addresses | 4,016 |
| IPv4 addresses | 4,063 |
| Phone numbers | 545 |
| Street addresses | 61 |
| City + state + ZIP | 22 |
| Tracking URLs | 0 |
| SSN-shaped | 0 |

**Known edge cases:** One malformed address pattern in the corpus
(`NNA@WA@winlink.org`, double-`@`) slipped through email redaction;
the surviving fragment is `WA@winlink.org`, which is the public
Winlink domain with too-short a local-part to be meaningfully
identifying. Otherwise, post-redactor scans found zero PII leaks.

To re-run the redactor and verify:

```sh
# Re-run on the original raw corpus (kept locally per no-disk-creds-default).
# Re-running on this already-redacted version is a no-op (sentinel strings
# are immune to the redactor's patterns).
python3 redact.py
sha256sum corpus.jsonl
```

## How to reproduce the theme-quantification table

```sh
python3 themes.py
# Writes themes.tsv to the same directory.
```

The script reads `corpus.jsonl`, applies the 23 keyword patterns from
the synthesis note against `subject + first-post-body-snippet`, and
counts the matches. The patterns are the **exact** ones used in the
synthesis; changes to the patterns should be discussed separately from
changes to the corpus.

**Methodology caveats for the theme percentages:**

- Token-based matching is approximate — a thread can match multiple
  themes; percentages do not sum to 100.
- Patterns match on subject + first-post-body snippet (first 500
  chars), not the full post body. Deeper analysis (full-body matching,
  reply-thread analysis) is possible by extending `themes.py` to
  consume more of the per-thread data.
- The percentages are within ±2% of the synthesis-note numbers because
  redaction nudged a few keyword counts (e.g., an email containing the
  word `net` was replaced with `[email-redacted]`, removing the `net`
  keyword from the EmComm bucket).

## What this corpus is, and what it is NOT

**This corpus IS:**

- A representative sample of recent activity in the Winlink Programs
  Group (the most-recent ~6 months to 1 year of front-page traffic,
  depending on the group's posting rate during the scrape window).
- Hard evidence for claims about recurring themes in the group's
  user-pain traffic.
- Reproducible from the committed scripts (with fresh cookies and a
  matching scrape window).

**This corpus is NOT:**

- A complete history of the Winlink Programs Group. The scrape hit
  Google's anti-automation gate at thread ~4,112; deeper history is
  not represented here.
- A statistical sample drawn from a defined population. It's a
  convenience sample of the most-recent threads visible from the
  authenticated session at scrape time. Inferences beyond "this many
  threads matched this pattern in this window" should be made
  carefully.
- A canonical Winlink-team-endorsed dataset. Winlink (ARSF) does not
  publish a research corpus; this is third-party-collected via
  legitimate read access.
- A substitute for talking to actual Winlink operators about their
  experiences. The corpus quantifies; it does not diagnose root
  causes.
